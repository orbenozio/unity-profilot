# Profilot - אפיון

> מסמך אפיון מאוחד. כל סקשן בבעלות סוכן אחד בלבד.
> כותבים רק את הסקשנים שלכם, מפנים לסקשנים של אחרים עם [[#section]], וקונפליקט עם מה שסוכן קודם כתב נרשם ביומן ה-Cross-cutting בתחתית - לא משכתבים עבודה של סוכן אחר.

**סטטוס:** טיוטה (DRAFT v2 - עבר גייט review ומעבר קוהרנטיות; נותרו החלטות בעלים פתוחות Q4/Q5/Q6/UX-Q1/UX-Q3)
**בעלים:** product / ux-designer / architect
**עודכן:** 2026-06-24

**הכרעת אריזה (נסגר עם המשתמש):** UPM package בתוך פרויקט Unity (tripwire + שליפת פריים דרך Editor APIs) + CLI חיצוני ש-Claude Code מריץ דרך Bash ומחזיר JSON מובנה ל-stdout. בלי MCP server ובלי VS Code extension - CLI יעיל ונוח יותר לסוכנים.

**הכרעות שנסגרו לסבב התיקון (DRAFT v2):** ההכרעות הבאות נסגרו על ידי הבעלים בעקבות גייט ה-review, וכל שכבה מחויבת להן:

1. **דמו מול live - מסגור.** הדמו (Flow A / שלב 0) הוא *מנוע אימוץ ויראלי על נתונים חלקיים*, לא המוצר. המוצר הוא ה-live (נתונים מובנים). screenshot הוא best-effort: ללא call hierarchy אמין, ולכן המיפוי marker->קוד (M3, JTBD-3) רלוונטי ל-live בלבד. קריטריוני הקבלה והמדדים מופרדים בין מסלול A למסלול B.
2. **מי סוגר אירוע + ערוץ feedback.** ה-Editor (החלון) הוא הבעלים של כתיבת ה-event store; ה-CLI נשאר read-only. בחלון Profilot יש כפתור ידני "סמן כנסקר / לא בעיה אמיתית", וזה גם ערוץ ה-feedback שעליו נשענים M2/M4/M9. זו דרישת מוצר, לא רק עיצוב.
3. **שפת ה-CLI (A-Q1).** Node.js + הפצת npm. ה-CLI הוא transport טהור (pass-through JSON), ולכן שיתוף DTOs עם ה-package כמעט לא רלוונטי, וקלות ההפצה מכריעה.
4. **גרסת Unity מינימלית (Q1).** Unity 6000.3 כמינימום (עודכן על ידי הבעלים 2026-07-02; קודם 2022.3 LTS). זו הגרסה שעליה פותח ואומת המוצר בפועל; ProfilerRecorder ו-HierarchyFrameDataView יציבים בה.
5. **ספי ברירת מחדל (Q2).** placeholder עד כיול: frame hitch מזוהה **יחסית ל-baseline מתגלגל של הפרויקט עצמו** (פריים מעל `baseline * 2.0` ומעל רצפה אבסולוטית של 12ms) ולא מול תקציב 60fps קבוע - כדי שהזיהוי יעבור בין פרויקטים ובין יעדי frame-rate בלי הצפת false positives (עודכן 2026-07-02, קודם 16.6ms קבוע), GC alloc per frame מעל 0, draw calls מעל סף יחסי ל-baseline. מותר לקבע אותם בקריטריוני הקבלה של JTBD-2 כ-placeholder.
6. **תיקון אוטומטי (Q4).** NG2 נשאר נחרץ לאיטרציה זו: מאבחנים ומציעים, לא מחילים. (Claude Code ממילא עורך לפי בקשת המשתמש.)
7. **הנחיית פרויקט.** ה-UPM package מספק קובץ הנחיה תבניתי שהמשתמש מעתיק/מקשר ל-CLAUDE.md ידנית (לא הזרקה אוטומטית באיטרציה זו). שייך לשלב 0.
8. **physics (NG5).** hitch שמקורו physics עדיין ייתפס כ-frame_hitch גנרי; האבחון רשאי לומר "המקור ב-physics, מחוץ לסקופ הניתוח הנוכחי".

## חלק א' - מוצר (בעלות: product)

**סטטוס שכבת המוצר:** DRAFT v2

### 1. בעיה והקשר

**הבעיה.** לקרוא את ה-Unity Profiler ואת ה-Frame Debugger ולהבין למה הפריים נתקע (frame hitch), למה מוקצה garbage כל פריים, או למה יש יותר מדי draw calls - זו מיומנות של מפתח סניור. הכלים האלה חושפים אלפי markers בעץ היררכי, בלי לומר מה חשוב ומה רעש, ובלי לקשר בין marker לשורת הקוד שאחראית לו. מי שלא עבר את עקומת הלמידה הזו פשוט לא יודע איפה להתחיל להסתכל.

**מי סובל מזה.** מפתחי Unity מתחילים-עד-בינוניים, שיודעים לכתוב gameplay אבל מפחדים מאופטימיזציה. עבורם ה-Profiler הוא קיר. התוצאה: או שהם מתעלמים מבעיות ביצועים עד שהמשחק כבר תקוע על מכשיר אמיתי וכואב לתקן, או שהם משקיעים שעות בניחושים (להשבית אובייקטים אקראית, לחפש בפורומים) בלי שיטה.

**הכאב הנוכחי.** אין כלי שלוקח את הידע הנדיר של "לקרוא פרופיילר" והופך אותו לזמין למפתח רגיל. כל הפתרונות הקיימים מניחים שאתה כבר יודע לקרוא את הנתונים - הם מציגים יותר נתונים, לא מסבירים אותם. הפער הוא לא בכלי המדידה (Unity כבר מודד מצוין), אלא בשכבת הפרשנות.

**הנחת היסוד שעליה הכל עומד: יש נתוני profiler אמיתיים.** המוצר כולו - בשני המסלולים - נשען על נתונים שמקורם בהרצת Play Mode אמיתית שבה Unity מדד את הפריים. בלי זה אין מוצר בשום מסלול. גם מסלול הדמו (screenshot) מניח שהמשתמש כבר היה ב-Play Mode בזמן הצילום: ה-screenshot הוא תיעוד של מדידה שכבר קרתה, לא תחליף לה. ה-markers האלה זמינים ב-Play Mode ב-Editor וב-development builds בלבד (ראה NG3) - וזה תקף לשני המסלולים, כולל הדמו, כי גם screenshot מצולם מתוך אותה הרצה.

**למה עכשיו.** שני דברים התחברו. ראשית, Unity חושפת את נתוני הפרופיילר כ-data מובנה דרך API (`ProfilerRecorder` מ-2020.2, ו-`ProfilerDriver` / `HierarchyFrameDataView` בצד Editor) - אפשר לקרוא frame time, GC alloc, draw calls ועץ markers מלא בלי אובדן מידע. שנית, מודל שפה כמו Claude מסוגל לקחת את ה-data המובנה הזה ביחד עם הקוד מהריפו ולהפיק אבחון ברמת סניור. הצומת הזה לא היה אפשרי לפני שנה.

**ההקשר התרבותי (מנוע האימוץ).** הקהל חופף לקהל סדרת היוטיוב "Claude Code for Unity" (פרסונה 3, ו-M10). מסלול הדמו נועד בדיוק לרכוב על זה: ראיה זולה ומהירה לערך שמושכת אנשים פנימה אל ה-live. הכאב אמיתי ויומיומי, והוא עובד על נתוני profiler וקוד שזהים בכל פרויקט - אין את בעיית ה"כל משחק שונה".

### 2. מטרות / לא-מטרות

**מטרות**

- G1. לתת למפתח שלא יודע לקרוא פרופיילר אבחון מובן בשפה אנושית של בעיית ביצועים אמיתית בפרויקט שלו, כולל הצבעה על האזור בקוד שאחראי (מיפוי הקוד רלוונטי למסלול ה-live; ראה הכרעה 1).
- G2. לתפוס בעיות ביצועים אוטומטית בזמן ריצה (tripwire), בלי שהמפתח יצטרך לדעת מתי או איפה להסתכל.
- G3. להוכיח ערך מהר ובמעט חיכוך: מסלול דמו שמייצר אבחון תוך דקות מבלי שום אינטגרציה בקוד, על בסיס נתונים חלקיים (best-effort) מתוך screenshot (ראה [[#5-scope---בפנים--בחוץ]]).
- G4. להבחין בין רעש לאות: לא להציף את המפתח בכל תנודה, אלא להתריע רק על בעיות אמיתיות ביחס לתקציב ביצועים מוגדר.
- G5. לאפשר השארה דולק ברקע ("שומר ביצועים שיושב בעורף") בעלות נמוכה - קריאה ל-LLM רק על אירוע אמיתי, לא כל פריים.

**לא-מטרות (במכוון לא עושים)**

- NG1. לא בונים פרופיילר משלנו ולא מחליפים את ה-Unity Profiler. אנחנו שכבת פרשנות מעל הנתונים שלו.
- NG2. לא מתקנים את הקוד אוטומטית ולא מבצעים merge בלי אישור. המוצר מאבחן ומציע; ההחלטה והעריכה אצל המפתח. זו החלטה נחרצת לאיטרציה זו (הכרעה 6) - לא שאלה פתוחה. מסלול apply בהסכמה נשקל רק כהרחבה עתידית (ראה Q4).
- NG3. לא מכסים בעיות ביצועים ב-release build. עץ ה-markers וה-call hierarchy זמינים ב-Play Mode ב-Editor וב-development builds בלבד; זה תואם את מקרה השימוש (מפתחים מפתחים ב-Editor) ולא ננסה לעקוף את זה. הערה לתפר: מגבלה זו חלה גם על מסלול הדמו, כי גם screenshot מצולם מתוך הרצת Play Mode.
- NG4. לא פותרים בעיות שאינן ביצועים (לוגיקת gameplay, באגים פונקציונליים, עיצוב). המיקוד הוא frame time, זיכרון/GC, ו-rendering (draw calls / batches).
- NG5. בגרסה הראשונה לא מנתחים ביצועי GPU לעומק ולא physics לעומק - תופסים מה ש-`FrameTimingManager` ו-counters בסיסיים נותנים בזול, והרחבה היא roadmap. hitch שמקורו physics עדיין ייתפס כ-frame_hitch גנרי, והאבחון רשאי לומר "המקור ב-physics, מחוץ לסקופ הניתוח הנוכחי" (הכרעה 8).
- NG6. לא MCP server ולא VS Code extension. האריזה נסגרה: UPM package + CLI חיצוני (ראה כותרת המסמך). זה מוזכר כאן רק כדי לסגור ציפיות, לא כהחלטת מוצר פתוחה.

### 3. משתמשים ופרסונות

**פרסונה 1 - "נועה, המתחילה שמפחדת מהפרופיילר" (הקהל המרכזי).**
מפתחת אינדי / סטודנטית / hobbyist. כותבת C# ו-gameplay בסבירות, אבל מעולם לא פתחה את ה-Frame Debugger ברצינות. כשהמשחק שלה מתחיל לקרטע היא לא יודעת אם זה ה-GC, ה-rendering או הקוד שלה. הקשר: עובדת לבד או בצוות קטן, בלי סניור שישב לידה. אילוץ: סף הסבלנות לכלי שדורש setup ארוך נמוך מאוד - אם זה לא נותן ערך בהרצה הראשונה, היא נוטשת. היא משתמשת ב-Claude Code כבר היום (חופפת לקהל היוטיוב).
ה-job המוביל שלה: JTBD-1 (דמו), ואחר כך JTBD-2/3/4 ב-live.

**פרסונה 2 - "איתי, הבינוני שיודע שיש בעיה אבל לא לאן לחפור".**
מפתח עם 2-4 שנות ניסיון. מכיר את המושגים (draw calls, GC alloc) ויודע לפתוח את ה-Profiler, אבל קריאת עץ ה-markers וקישורו לקוד עדיין איטית וכואבת לו. הקשר: יש לו פרויקט אמיתי בגודל בינוני, רוצה לקצר את הלולאה מ"יש hitch" ל"זו השורה". אילוץ: לא יסבול false positives - אם הכלי מתריע על רעש הוא מאבד אמון מהר.
ה-job המוביל שלו: JTBD-3 (אבחון עם הקשר קוד) ב-live, ו-JTBD-5 (שקט).

**פרסונה 3 - "ילן, יוצר התוכן" (משני, מניע אימוץ).**
מפתח/קריאייטור שמייצר תוכן Unity + Claude Code. מחפש רגעי "וואו" קצרים וברורים להדגמה. הקשר: צריך שהמסלול מ-input ל-אבחון מרשים יהיה קצר, ויזואלי, וניתן להקלטה. אילוץ: צריך שהפלט יהיה קריא ומשכנע גם לצופה שאינו מומחה.
ה-job המוביל שלו: JTBD-7 - "שהאבחון יהיה מובן ומשכנע גם לצופה שאינו מומחה, כדי שאוכל להפוך אותו לתוכן". נמדד דרך M10.

המכנה המשותף: כולם עובדים ב-Editor, כולם כבר בעולם Claude Code, וכולם נמדדים על "כמה מהר הגעתי מבעיה מעורפלת לכיוון פעולה ברור".

### 4. User stories / jobs-to-be-done

הניסוח הוא jobs (צורך), לא features. לכל job יש קריטריוני קבלה שניתן לבדוק. שימו לב להפרדה בין מסלול דמו (A, best-effort) למסלול live (B): מיפוי marker->קוד מובטח רק ב-live.

**JTBD-1 (מסלול דמו / screenshot - best-effort).** כשיש לי spike בפרופיילר ואני לא מבין אותו, אני רוצה לקבל אבחון מהיר ממה שכבר מוצג לי, כדי להבין מה קורה בלי להתקין כלום.
- קבלה: המשתמש מספק נתוני פריים (screenshot של ה-Profiler או טקסט/תיאור, שצולמו מתוך הרצת Play Mode) ומקבל אבחון בשפה אנושית של הגורם הסביר, בלי שום שינוי בקוד הפרויקט.
- קבלה: המסלול הזה עובד גם בלי ה-UPM package מותקן (input ידני / חיצוני).
- קבלה: כשהנתונים ב-screenshot חלקיים (אין call hierarchy אמין), האבחון יוצא על בסיס מה שכן נקרא, ומציין מפורשות שהוא best-effort על נתונים חלקיים - לא מתחייב על מיפוי לקוד (זה JTBD-3, ל-live בלבד).

**JTBD-2 (tripwire live).** כשאני משחק את המשחק שלי ב-Editor, אני רוצה שהכלי יתפוס לבד את הפריים הבעייתי, כדי שלא אצטרך לדעת מתי להסתכל או איך לשחזר את הרגע.
- קבלה: כשמתרחשת אנומליה ביחס לתקציב מוגדר, המערכת מסמנת אירוע אוטומטית בלי פעולה יזומה של המשתמש. ספי הברירת-מחדל לבדיקה (placeholder עד כיול, הכרעה 5): frame time מעל 16.6ms (60fps); GC alloc בפריים מעל 0 בייט; draw calls מעל סף יחסי ל-baseline נע.
- קבלה: הניטור רץ ברקע במהלך Play Mode בלי שהמשתמש מפעיל אותו ידנית בכל פעם.

**JTBD-3 (אבחון עם הקשר קוד - live בלבד).** כשנתפס אירוע live, אני רוצה אבחון שמקשר בין ה-marker לקוד שלי, כדי שאדע איפה לפתוח את הקובץ ולא רק "יש בעיה איפשהו".
- קבלה: הפלט כולל את הגורם המאובחן + הפניה לאזור הקוד הרלוונטי (קובץ/סימבול ככל שניתן למפות).
- קבלה: כשלא ניתן למפות marker לקוד, הפלט אומר זאת במפורש במקום להמציא מיקום.
- הערה: ה-job הזה אינו חל על מסלול הדמו (הכרעה 1), כי ל-screenshot אין call hierarchy אמין.

**JTBD-4 (תיקון מוצע).** אחרי שהבנתי מה הבעיה, אני רוצה הצעת תיקון קונקרטית, כדי שאדע מה לעשות הלאה ולא רק שיש לי בעיה.
- קבלה: הפלט כולל לפחות צעד פעולה אחד קונקרטי או הצעת שינוי קוד.
- קבלה: התיקון מוצג כהצעה לאישור, לא מוחל אוטומטית (NG2, נחרץ לאיטרציה זו).

**JTBD-5 (שקט כברירת מחדל).** כשאין בעיות אמיתיות, אני רוצה שהכלי ישתוק, כדי שאוכל להשאיר אותו דולק בלי הסחה ובלי הצפה.
- קבלה: כשכל המדדים בתוך התקציב, לא נוצרים אירועים ולא נשלחות קריאות LLM.
- קבלה: למשתמש יש דרך להגדיר את ספי התקציב (אחרת ברירת המחדל מהכרעה 5).

**JTBD-6 (עלות נשלטת).** כמשתמש שמשאיר את הכלי דולק, אני רוצה שלא יבזבז קריאות LLM, כדי שזה יישאר זול מספיק להפעלה רציפה.
- קבלה: קריאת LLM מתרחשת רק על אירוע שעבר את סף ה-tripwire, לא לפי פריים/שעון.
- קבלה: יש מנגנון שמונע מבול קריאות מאותו אירוע חוזר (דה-דופ / קצב מקסימלי - הגדרה טכנית ב-[[#15-nfrs---ביצועים--אבטחה--scale]]).

**JTBD-7 (אבחון כתוכן - פרסונה 3).** כיוצר תוכן, אני רוצה שהאבחון יהיה מובן ומשכנע גם לצופה שאינו מומחה, כדי שאוכל להפוך רגע אבחון לפרק/פוסט/קליפ.
- קבלה: הפלט קריא בשפה אנושית בלי ז'רגון בלתי-מוסבר, עם מבנה עקבי (סוג בעיה -> אבחון -> ראיות -> תיקון) שקל לקרוא על מסך בהקלטה.
- קבלה: רגע "מבעיה מעורפלת לכיוון פעולה ברור" ניתן להדגמה מקצה לקצה תוך דקות (חופף ל-M1 ול-M10).

**JTBD-8 (סגירת אירוע ומשוב - מנגנון שעליו נשענים המדדים).** כשהבנתי אירוע, אני רוצה לסמן אותו כנסקר או כ"לא בעיה אמיתית", כדי שהתור יישאר נקי ושהכלי ילמד מה היה רעש.
- קבלה: בחלון Profilot יש כפתור ידני "סמן כנסקר / לא בעיה אמיתית" (הכרעה 2). זו דרישת מוצר, לא רק עיצוב; פירוט ה-UX ב-[[#10-התנהגות-אינטראקציה-ו-edge-cases]].
- קבלה: סימון "לא בעיה אמיתית" נשמר כ-feedback, וזהו הקלט שעליו נשענים M2 (מובן/עזר), M4 (false positives) ו-M9 (מעבר דמו->live).
- קבלה: ה-Editor (החלון) הוא הבעלים של כתיבת ה-event store; ה-CLI נשאר read-only (תפר ל-[[#12-ארכיטקטורת-מערכת]] ו-[[#14-מודל-נתונים-ו-apis]]).

### 5. Scope - בפנים / בחוץ

הסקופ בנוי על ההתקדמות מ-דמו ל-live. שני המסלולים בפנים לאיטרציה הזו, כי הדמו הוא מנוע האימוץ (על נתונים חלקיים) וה-live הוא המוצר (נתונים מובנים).

**בפנים (איטרציה ראשונה)**

- מסלול A - דמו (best-effort): קבלת נתוני פריים מ-input חיצוני/ידני (screenshot של הפרופיילר או טקסט) -> אבחון בשפה אנושית. אפס אינטגרציה בקוד. ללא התחייבות למיפוי marker->קוד (אין call hierarchy אמין ב-screenshot).
- מסלול B - live tripwire: ניטור רציף ב-Play Mode עם היוריסטיקה מקומית שתופסת אנומליות, ושליפת נתוני הפריים המלא בעת אירוע. כולל מיפוי marker->אזור קוד והצעת תיקון.
- ערוץ feedback ידני בחלון ("סמן כנסקר / לא בעיה אמיתית", הכרעה 2) שעליו נשענים M2/M4/M9.
- שלושה צירי בעיה: frame time / hitches, GC allocations, ו-rendering (draw calls / batches / set-pass).
- ספי תקציב הניתנים להגדרה, עם ברירות מחדל (placeholder, הכרעה 5).
- פלט מובנה הניתן לצריכה אוטומטית על ידי סוכן. דרישת המוצר: הפלט צריך להיות parse-able על ידי Claude Code בלי התערבות אנושית. המימוש הטכני (JSON ל-stdout מה-CLI, exit codes, סכמה) ב-[[#12-ארכיטקטורת-מערכת]] ו-[[#14-מודל-נתונים-ו-apis]].
- קובץ הנחיית פרויקט תבניתי שהמשתמש מעתיק/מקשר ל-CLAUDE.md ידנית (הכרעה 7).

**בחוץ (לא באיטרציה הזו)**

- תמיכה ב-release build / נתוני production מהשטח (חסם API - NG3).
- ניתוח GPU לעומק ו-physics profiling לעומק (roadmap - NG5).
- תיקון אוטומטי / merge בלי אישור (NG2, נחרץ לאיטרציה זו; apply בהסכמה הוא הרחבה עתידית - Q4).
- מיפוי marker->קוד במסלול הדמו (best-effort בלבד שם; המיפוי המובטח הוא ל-live - הכרעה 1).
- הזרקה אוטומטית של הנחיית הפרויקט (באיטרציה זו ההעתקה ידנית - הכרעה 7).
- אינטגרציות הפצה מעבר ל-UPM + CLI: MCP server, VS Code extension (NG6).
- ניתוח ביצועים שאינו frame/memory/render (networking, asset load times, build size) - לא באיטרציה זו.
- שיתוף פעולה צוותי / היסטוריית אירועים מתמשכת / דשבורד - לא באיטרציה זו.

**הערת תפר:** עיצוב ההצגה של כל מצב (אין בעיות / נתפס spike / מחכה לקלוד / הוצע fix) וכפתור ה-feedback הוא בבעלות ux-designer ב-[[#9-מסכים-ו-states-עיקריים]] ו-[[#10-התנהגות-אינטראקציה-ו-edge-cases]]. שכבת המוצר רק דורשת שהמצבים יהיו ניתנים להבחנה, שמצב "שקט" יהיה ברירת המחדל, ושקיים ערוץ feedback ידני (JTBD-8).

### 6. מדדי הצלחה

המדדים נגזרים מהמטרות. מספרי היעד הם השערת עבודה לאיטרציה ראשונה וניתנים לכיול אחרי קליפ/בטא ראשונה. **מקור ה-feedback לכל מדד התנהגותי (M2/M4/M9) הוא כפתור ה-feedback הידני מ-JTBD-8** - בלעדיו אין דרך לאסוף את הנתון. שימו לב להפרדה דמו (A) מול live (B).

**אקטיבציה וערך ראשון - מסלול דמו A (G3, פרסונה 1, פרסונה 3)**

- M1. Time-to-first-diagnosis במסלול הדמו: מ-input ועד אבחון מוצג מתחת ל-5 דקות, ללא שינוי קוד. יעד: 90% מההרצות.
- M2. שיעור אבחונים שהמשתמש מסמן כ"מובן / עזר לי להבין" (דרך כפתור ה-feedback, JTBD-8): יעד מעל 70% במדגם בטא. נמדד בשני המסלולים בנפרד.

**איכות האבחון - מסלול live B (G1, G4)**

- M3. דיוק מיפוי marker -> קוד, **ב-live בלבד** (לא רלוונטי לדמו - הכרעה 1). שיטת מדידה: סט קבוע של "תרחישי-זהב" - תרחישים מתוכננים שבהם הקוד האחראי ידוע מראש (GC כל פריים מ-`new` ב-Update, ספייק draw calls מ-instancing שבור, hitch מ-load סינכרוני, וכו'). הדנומינטור הוא מספר תרחישי-הזהב; המונה הוא אלה שבהם האבחון הצביע על האזור הנכון. יעד מעל 60% (שאפתני, ייכויל). מאוחד עם M5 על אותו סט תרחישי-זהב (M5 בודק תפיסה, M3 בודק דיוק המיפוי).
- M4. שיעור false positives ב-tripwire (אירועים שהמשתמש סימן כ"לא בעיה אמיתית" דרך כפתור ה-feedback, JTBD-8): יעד מתחת ל-15% מהאירועים. **ל-live בלבד.**
- M5. recall על סט תרחישי-הזהב (אותו סט של M3): מתוך התרחישים המתוכננים, המערכת תופסת לפחות 80%. **ל-live בלבד.**

**שתיקה ועלות (G4, G5)**

- M6. רעש בזמן שקט: במהלך הרצת live "נקייה" (כל המדדים בתקציב), אפס קריאות LLM ואפס אירועים. בינארי, חייב להתקיים.
- M7. עלות LLM ממוצעת לשעת פיתוח עם הכלי דולק: יעד שיישאר נמוך מספיק להשארה רציפה (סף כספי מדויק - ראה Q5).
- M8. overhead הניטור על frame time ב-Play Mode: דרישת המוצר היא **מתחת ל-0.5ms לפריים בזמן שקט** (כ-3% מתקציב 16.6ms). הערך המדויק והמדידה ב-[[#15-nfrs---ביצועים--אבטחה--scale]].

**אימוץ ותוכן (G3, פרסונה 3)**

- M9. שיעור משתמשים שעוברים ממסלול הדמו (A) להתקנת ה-live (B) ולפחות אבחון live אחד מסומן בכפתור ה-feedback: יעד מעל 25% - זה המדד שמראה שהדמו באמת מזין את המוצר.
- M10. מספר אבחונים שהופכים לתוכן/שיתוף (פרק, פוסט, קליפ) - מדד ויראליות, ייאסף ידנית בתחילת הדרך. קשור ל-JTBD-7 ולפרסונה 3.

**שאלות פתוחות לבעלים (החלטות שרק המשתמש יכול לקבל)**

- Q4. האם, **בהרחבה עתידית** (לא באיטרציה זו - NG2 נחרץ), לפתוח מסלול apply בהסכמה (Claude מחיל את ה-diff אחרי אישור מפורש), או להישאר ב"הצעה בלבד" לתמיד.
- Q5. מהו סף העלות הכספי שמתחתיו נחשב "זול מספיק להשארה דולק" (קלט ל-M7).
- Q6. שם/דומיין/handle - האם Profilot ננעל סופית (תלוי בבדיקת דומיין ו-publisher לפי ה-brief), כי זה משפיע על המיתוג בפלט וגם על שם חבילת ה-npm (ההכרעה על Node/npm כבר נשענת על כך ש-npm פנוי).

> הערה: Q1 (גרסת Unity), Q2 (ספי ברירת מחדל) ו-Q3 (סדר עדיפות A מול B) נסגרו בבלוק ההכרעות (4, 5, 1 בהתאמה) והוסרו מכאן.
> שאלות UX פתוחות (ממוספרות בשכבת ה-UX): UX-Q1 (השהיית Play אוטומטית), UX-Q3 (קיצור מקלדת גלובלי). UX-Q2 נסגר ב-[[#15-nfrs---ביצועים--אבטחה--scale]].

## חלק ב' - חוויית משתמש (בעלות: ux-designer)

**סטטוס שכבת ה-UX:** DRAFT v2

**עיקרון מנחה לכל החלק.** ל-Profilot אין UI משלו ואין חלון מומצא. החוויה חיה בשני בתים בלבד, כפי שנסגר באריזה ([[#חלק-א---מוצר-בעלות-product]] כותרת המסמך): (1) **Unity Editor** - שם רץ ה-tripwire וצף האות הראשוני ("נתפס spike"), דרך משטחים מקוריים של Unity (חלון עורך קטן + Console); (2) **Claude Code** - שם נצרך הפלט המובנה של ה-CLI, שם מוצג האבחון וההצעה לתיקון, ושם המפתח מחליט ומבצע. כל "מסך" בסעיף 9 הוא אחד משני המשטחים האלה - לא ממשק חדש. הכלל: Unity אחראי על *גילוי* ו*איסוף*, Claude Code אחראי על *פרשנות* ו*פעולה*. המעבר בין השניים הוא הרגע הקריטי בחוויה, ועליו נעבוד הכי קשה.

### 7. User flows ומסעות

כל job מ-[[#4-user-stories--jobs-to-be-done]] ממופה כאן ל-flow קונקרטי. נסמן את המשטח של כל צעד: **[Editor]** או **[CC]** (Claude Code).

#### Flow A - מסלול דמו / screenshot (משרת JTBD-1, JTBD-7; וגם JTBD-4 כ-best-effort)

המסלול שמוכיח ערך תוך דקות, בלי שום אינטגרציה בקוד ובלי שה-UPM package בהכרח מותקן. (הערה: JTBD-3 - מיפוי marker->קוד מובטח - הוא ל-live בלבד, הכרעה 1; בדמו זה best-effort שמסויג מפורשות.)

1. **[CC]** המפתח נתקל ב-spike, פותח את ה-Profiler של Unity, ומצלם screenshot (או מעתיק טקסט מה-Profiler / כותב תיאור ידני).
2. **[CC]** המפתח גורר/מצרף את ה-screenshot ל-Claude Code ומבקש אבחון (למשל "תסתכל על זה, למה יש לי spike"). אין צורך בפקודה מיוחדת - Claude Code כבר יודע, דרך הנחיות הפרויקט, שהכלי הזה זמין.
3. **[CC]** Claude קורא את הנתונים מה-screenshot, ובמקביל יכול לקרוא את הקוד בריפו (Claude Code כבר בתוך הפרויקט).
4. **[CC]** מוצג **אבחון בשפה אנושית**: מה הגורם הסביר (frame time / GC / rendering), ואם ניתן - הפניה לאזור הקוד.
5. **[CC]** מוצגת **הצעת תיקון** קונקרטית (צעד פעולה או diff מוצע), כהצעה לאישור - לא מוחל אוטומטית.
6. **[CC]** המפתח מחליט: לאמץ, לבקש חידוד, או לדחות.

נקודות כשל ומענה: screenshot מטושטש / חתוך / לא של Profiler -> ראה edge cases ב-[[#10-התנהגות-אינטראקציה-ו-edge-cases]]. אין מבוי סתום: גם כש-Claude לא מצליח למפות לקוד, הוא חייב להחזיר אבחון מבוסס-נתונים + לומר מפורשות שלא מצא מיפוי (JTBD-3 קבלה שנייה).

הערת תפר: צורת הקריאה (screenshot מול טקסט) ומה שניתן לחלץ מ-screenshot היא שאלה טכנית ל-[[#14-מודל-נתונים-ו-apis]]; כאן רק מובטח שלמשתמש יש מסלול אחד פשוט שלא דורש התקנה.

#### Flow B - live tripwire (משרת JTBD-2, JTBD-3, JTBD-4, JTBD-5, JTBD-6, JTBD-8)

המוצר האמיתי - "שומר ביצועים שיושב בעורף". מתחלק ל-flow התקנה חד-פעמי ו-flow תפעולי חוזר.

**B0 - התקנה (חד-פעמי):**
1. **[Editor]** המפתח מתקין את ה-UPM package (דרך Package Manager / Git URL).
2. **[Editor]** נפתח חלון Profilot (תפריט `Window > Profilot`), שמראה שהניטור מזוהה ופעיל, ומאפשר לכוונן ספי תקציב (ברירת מחדל סבירה כבר טעונה - placeholder לפי הכרעה 5).
3. **[CC]** המפתח מעתיק/מקשר ידנית את קובץ הנחיית הפרויקט ל-CLAUDE.md (לא הזרקה אוטומטית באיטרציה זו - הכרעה 7), כך ש-Claude יודע להריץ את ה-CLI. צעד זה הוא תפר ל-[[#12-ארכיטקטורת-מערכת]].

**B1 - תפעול שוטף (החוויה החוזרת):**
1. **[Editor]** המפתח לוחץ Play ומשחק/בודק כרגיל. Profilot דוגם ברקע, שקט (state "אין בעיות", ראה 9).
2. **[Editor]** מתרחשת אנומליה. ה-tripwire תופס ומסמן trip, וה-Editor מתחיל **לשלוף את הפריים המלא** ברקע. כל עוד הפריים לא persisted, האירוע מופיע ב-state ביניים **"נתפס - בשליפה"** ולא ניתן עדיין לאבחון (ראה 9). המשחק לא נעצר.
3. **[Editor]** השליפה הסתיימה והאירוע נכתב אטומית ל-event store. עכשיו הוא ב-state **"נתפס - מוכן לאבחון"**: צף **אות לא-חוסם** (שורה ב-Console + סימון בחלון Profilot). רק במצב הזה מוצגים מזהה האירוע ופקודת האבחון - כדי שהמשתמש לא יבקש אבחון על אירוע חלקי. (עצירה אוטומטית כן/לא - ראה UX-Q1 בסוף סעיף 7.)
4. **[Editor -> CC]** **המעבר.** האות מציין למפתח שיש אירוע ממתין, והדרך לאבחנו היא דרך Claude Code. שתי דרכים נתמכות: (א) המפתח אומר ל-Claude Code "תאבחן את האירוע האחרון" / Claude מריץ את ה-CLI שמושך את האירוע האחרון; (ב) המפתח מעתיק מזהה אירוע קצר מהחלון. ברירת המחדל היא הדרך הפשוטה - "האירוע האחרון" (הגדרת "האחרון" - ראה הערה למטה).
5. **[CC]** Claude מריץ את ה-CLI דרך Bash. ה-CLI מחזיר את ה-JSON של האירוע ב-ms (האירוע כבר persisted - אין כאן שליפה מ-Unity), קורא את הקוד הרלוונטי, ומבצע את **האבחון** (קריאת קוד + LLM - זה ה-progress הארוך, ראה state "מחכה לקלוד", 9).
6. **[CC]** מוצגת **הצעת תיקון** (state "הוצע fix"), כהצעה לאישור.
7. **[CC]** המפתח מחליט: לאמץ / לחדד / לדחות.
8. **[Editor]** **סגירת הלולאה (JTBD-8).** המפתח חוזר לחלון Profilot ולוחץ על הכפתור **"סמן כנסקר / לא בעיה אמיתית"** על האירוע. ה-Editor כותב את ה-feedback ל-event store (ה-Editor הוא הבעלים של הכתיבה; ה-CLI נשאר read-only - הכרעה 2). סימון "לא בעיה אמיתית" הוא הקלט ל-M4, וסימון "נסקר/עזר" מזין את M2/M9. החלון חוזר ל-state שקט אם אין אירועים נוספים בתור.

**"האירוע האחרון" - הגדרה.** "האחרון" = האירוע עם ה-timestamp (`capturedAt`) המאוחר ביותר ב-event store; שובר-שוויון לפי `frameIndex` ואז לפי `eventId`. זהו האירוע שמסך 2 מסמן כברירת מחדל לאבחון (ראה 9). הסכמה הטכנית של `latest.json` ב-[[#14-מודל-נתונים-ו-apis]].

**אין מבוי סתום:** כל אירוע שנתפס ניתן לאבחון או לסגירה ידנית (כפתור "סמן כנסקר"). אירוע שלא אובחן נשאר בתור ולא נעלם בשקט.

#### Flow C - שתיקה (משרת JTBD-5, JTBD-6)

זה ה-"flow" של אי-קרות. במהלך Play Mode נקי: אפס אירועים, אפס שורות Console, אפס קריאות LLM. החלון מציג state "אין בעיות" עם חיווי דיסקרטי שהניטור פעיל (כדי שהמשתמש יידע שהוא דולק, ולא יחשוב שהוא תקוע). זהו ברירת המחדל המוחלטת - ראה M6.

#### Flow D - אין נתונים זמינים (אזהרת ה-brief)

קריטי לחוויה: ה-markers זמינים רק ב-Play Mode ב-Editor וב-development builds (NG3). יש להבהיר זאת אקטיבית כדי לא להפתיע:
1. **[Editor]** המפתח פותח את חלון Profilot ב-**Edit Mode** (לא Play) -> החלון מציג state "אין נתונים - היכנס ל-Play Mode" (ראה 9), לא מסך ריק ולא שגיאה.
2. **[Editor]** המפתח מנסה להשתמש בכלי על build רגיל (לא development) -> הסבר מפורש שה-markers נחתכו וזה צפוי, עם הפניה לאיך לבנות development build.
3. **[CC]** המפתח מבקש מ-Claude לאבחן אבל אין אירוע / Play לא רץ -> ה-CLI מחזיר JSON עם סטטוס "no_data" ו-Claude מסביר בעברית פשוטה למה ומה לעשות, במקום לטעות.

**שאלה פתוחה שעלתה ב-flows (לבעלים):**
- UX-Q1. בעת תפיסת spike ב-Play Mode - האם להציע/לבצע השהיית Play אוטומטית כדי "להקפיא" את הרגע, או להשאיר את הריצה רציפה ולסמוך על השליפה שכבר נשמרה? (משפיע ישירות על ה-state "נתפס spike".)

> הערה: UX-Q2 (גודל תור והתנהגות בהתמלאות) נסגרה ב-[[#15-nfrs---ביצועים--אבטחה--scale]] - תקרת תור ~20, החדש דוחק את הישן ביותר שלא נסקר או מתאחד בדה-דופ.

### 8. ארכיטקטורת מידע (IA)

אין כאן אתר או אפליקציה עם ניווט עמוק. ה-IA הוא מיפוי של *איזה מידע חי איפה* ו*איך זורמים ביניהם*. שני עוגנים, ומאגר אירועים ביניהם.

```
                 +---------------------------------------------+
                 |            Profilot (UPM package)           |
                 |  tripwire דוגם ברקע, תופס, שולף פריים מלא    |
                 +---------------------------------------------+
                          |                         ^
            כותב אירועים  v                         | קורא אירוע אחרון/לפי id
                 +---------------------+      +---------------------+
   [Editor]      |  חלון Profilot      |      |   CLI (Bash)        |   [CC]
   בית הגילוי    |  + שורת Console     |----->|   JSON ל-stdout     |---> Claude Code
                 +---------------------+      +---------------------+        בית הפרשנות
                          |                                                       |
                          +--- מאגר אירועים משותף (event store) -------------------+
```

**עוגן 1 - חלון Profilot ב-Editor (`Window > Profilot`).** משטח דק, לא לוח בקרה. תוכן, מלמעלה למטה לפי חשיבות:
1. **חיווי סטטוס** - פעיל / לא ב-Play / אין נתונים. תמיד גלוי, השורה הראשונה.
2. **אירוע נוכחי / אחרון** - אם נתפס spike: כותרת קצרה (סוג + חומרה + זמן), מזהה אירוע ניתן להעתקה, וכפתור **"סמן כנסקר / לא בעיה אמיתית"** (JTBD-8) שדרכו ה-Editor כותב feedback ל-store.
3. **תור אירועים** - רשימה קצרה של אירועים ייחודיים שנתפסו ולא נסקרו (אם יש יותר מאחד; אירוע מאוחד מוצג עם מונה ×N).
4. **ספי תקציב** - אזור הגדרות מתקפל (frame budget, GC, draw calls). סגור כברירת מחדל; זו לא הפעולה היומיומית.
5. **רמז פעולה** - שורת עזר קבועה: "לאבחון, בקש מ-Claude Code לאבחן את האירוע האחרון". זה הגשר ל-CC.

**עוגן 2 - Claude Code (סשן צ'אט + פלט CLI).** אין לנו שליטה על ה-chrome של Claude Code, ולכן ה-IA כאן הוא *מבנה הפלט* שאנו מייצרים, שחייב להיות עקבי וניתן לסריקה. סדר קבוע לכל אבחון (ראה 9, מסך 4):
1. שורת כותרת - סוג הבעיה + חומרה.
2. אבחון - מה קורה ולמה, בשפה אנושית.
3. ראיות - המספרים מהפריים (frame time, GC alloc, draw calls, ה-marker הדומיננטי).
4. מיקום בקוד - קובץ/סימבול, או הצהרה מפורשת שאין מיפוי.
5. הצעת תיקון - צעד פעולה או diff.
6. צעד הבא - מה אפשר לבקש עכשיו (לאמץ / לחדד / להתעלם).

**ניווט בין השניים:** חד-כיווני בעיקרו. Editor -> CC דרך "האירוע האחרון" או מזהה אירוע. CC -> Editor רק עקיף (Claude יכול להציע לפתוח קובץ, אבל לא שולט בחלון). אין סנכרון דו-כיווני באיטרציה זו - זו החלטה מודעת לפשטות. סגירת האירוע ("נסקר") היא פעולה ידנית **בחלון ה-Editor** (לא ב-CC), כי ה-Editor הוא הבעלים של ה-store.

**היררכיית מידע גלובלית:** סטטוס "האם דולק ובריא" > "יש אירוע ממתין" > "מה האירוע" > "הגדרות". המשתמש צריך לדעת בלי לחשוב את התשובה לשאלה "האם זה עובד עכשיו ויש משהו שדורש את תשומת לבי".

### 9. מסכים ו-states עיקריים

כל "מסך" הוא משטח מקורי (Unity או Claude Code), מתואר כ-wireframe טקסטואלי. כל ה-states שה-brief דורש מופיעים כאן במפורש ומסומנים.

#### מסך 1 - חלון Profilot ב-Editor, state "אין בעיות / שקט" [STATE 1]

הדיפולט. רוב הזמן המשתמש רואה את זה.

```
+--------------------------------------------------+
|  Profilot                                   [⚙]  |
+--------------------------------------------------+
|  ● פעיל · מנטר ב-Play Mode                        |   <- חיווי ירוק דיסקרטי
|                                                  |
|        אין בעיות שנתפסו                           |   <- מסר מרכזי שקט
|        הכל בתוך התקציב.                           |
|                                                  |
|        frame: 11.2ms / 16.6ms                    |   <- מדדים חיים, קטנים, אופציונלי
|        GC: 0 B/frame · draw calls: 142           |
|                                                  |
|  ────────────────────────────────────────────   |
|  ▸ ספי תקציב (מתקפל)                              |
+--------------------------------------------------+
```
חשוב: שקט אמיתי. בלי אנימציות מהבהבות, בלי badge אדום. החיווי הירוק והמספרים החיים הם רק כדי לבסס "זה דולק ובריא", מענה לחשש שהכלי תקוע.

#### מסך 1ב - אזור ספי התקציב פתוח [הרחבה של מסך 1/2]

זו הרחבה של מסך 1 או 2 כשה-toggle "ספי תקציב" נפתח (אותו אזור, לא חלון נפרד). שלושה שדות עם ברירות מחדל מהכרעה 5:

```
|  ▾ ספי תקציב                                     |
|     frame budget   [ 16.6 ] ms   (60fps)         |   <- מספרי, חיובי
|     GC alloc/frame [ 0    ] B    (0 = כל הקצאה)   |   <- שלם אי-שלילי
|     draw calls     [ ×1.5 ] מ-baseline נע        |   <- מכפיל יחסי, > 1.0
|     [ שמור ]   [ אפס לברירת מחדל ]                |
```
Validation: ערך לא-מספרי או שלילי -> הודעת שגיאה inline מתחת לשדה, כפתור "שמור" מנוטרל עד תיקון; frame budget = 0 נדחה ("חייב להיות גדול מ-0"). שינוי סף שמעלה את הרף מציג feedback ("הסף עלה - אירועים קלים יותר לא יתפסו", ראה 10). מספרים מוצגים LTR גם בתוך החלון העברי.

#### מסך 2 - חלון Profilot, state "נתפס spike" [STATE 2]

```
+--------------------------------------------------+
|  Profilot                                   [⚙]  |
+--------------------------------------------------+
|  ● פעיל · נתפס אירוע                              |   <- חיווי משתנה (כתום/ענבר, לא אדום בוהק)
|                                                  |
|  ⚠ Frame hitch · 48.3ms        « האחרון »        |   <- כותרת אירוע: סוג + חומרה; מסומן כברירת מחדל לאבחון
|     נתפס לפני 3 שניות · ⁨frame #5821⁩              |   <- frame number ב-bidi isolation
|     event id:  ⁨evt_5821_hitch⁩     [העתק]         |   <- מזהה ב-bidi isolation, ניתן להעתקה
|                                                  |
|  לאבחון: בקש מ-Claude Code                        |
|  "תאבחן את האירוע האחרון"             [העתק פקודה]|   <- הגשר ל-CC
|                                                  |
|  [ ✓ סמן כנסקר / לא בעיה אמיתית ]                 |   <- JTBD-8: ה-Editor כותב feedback ל-store
|                                                  |
|  ────────────────────────────────────────────   |
|  תור (2):  GC spike · 1.4MB   |  draw calls 980  |   <- אירועים ייחודיים; מאוחד מוצג עם ×N
+--------------------------------------------------+
```
חיווי במקביל ב-Console: שורת log אחת לא-חוסמת, למשל `[Profilot] Frame hitch 48.3ms caught at frame 5821 - ask Claude Code to diagnose (evt_5821_hitch)`. שורה אחת בלבד לאירוע, לא ספאם. אירוע חוזר זהה לא מוסיף שורות - הוא מעדכן מונה (`Frame hitch ×47`).

הערה: מונה **"תור (N)" = מספר האירועים הייחודיים** (אחרי דה-דופ), לא מספר ה-trips הגולמי. אירוע שאוחד מ-N trips מוצג כשורה אחת עם `×N`. הכפתור "סמן כנסקר" סוגר את האירוע (מסיר מהתור / מסמן feedback) - ראה Flow B שלב 8 ו-[[#10-התנהגות-אינטראקציה-ו-edge-cases]].

#### מסך 2א - חלון Profilot, state ביניים "נתפס - בשליפה" [STATE 2-pre]

הפריים נתפס אבל עוד לא נשלף/persisted. מובחן ממסך 2 כדי שהמשתמש לא יבקש אבחון על אירוע חלקי.

```
+--------------------------------------------------+
|  ● פעיל · נתפס אירוע                              |
|                                                  |
|  ⚠ Frame hitch · ⁨frame #5821⁩                     |
|     שולף נתוני פריים מלא...        ⠋              |   <- spinner; אין עדיין event id ולא פקודת אבחון
|                                                  |
|  (האבחון יהיה זמין כשהשליפה תסתיים)               |
+--------------------------------------------------+
```
מזהה האירוע ופקודת האבחון **אינם** מוצגים עדיין, וכפתור "סמן כנסקר" מנוטרל - אין על מה לתת feedback. ברגע שהשליפה מסתיימת המסך מתחלף למסך 2 ("נתפס - מוכן לאבחון"). אם השליפה נכשלת -> state שגיאה (ראה 10).

#### מסך 3 - Claude Code, state "מחכה לקלוד / אבחון בתהליך" [STATE 3]

זה מה שהמשתמש רואה ב-Claude Code בין הרצת ה-CLI לבין האבחון המוכן. ה-progress הארוך כאן הוא **האבחון** (קריאת הקוד + LLM), לא הרצת ה-CLI עצמה - ה-CLI רק קורא קובץ persisted ומחזיר JSON ב-ms.

```
> תאבחן את האירוע האחרון

  מריץ profilot diagnose --last ...
  ✓ נמשך אירוע ⁨evt_5821_hitch⁩ (frame hitch, 48.3ms)   [ms - קריאת קובץ]
  ✓ קורא את עץ ה-markers ואת הקוד הרלוונטי
  ⠋ מנתח ...                                          [כאן הזמן - LLM]
```
עקרון: progress גלוי וכן. כל צעד שמסתיים מקבל ✓, הצעד הפעיל מקבל spinner. ה-spinner **תמיד** מתחלף ל-✓ (הצליח) או ל-✕ (נכשל) - לעולם לא נשאר תלוי. המשתמש אף פעם לא מול מסך קפוא בלי לדעת אם משהו קורה. אם האבחון איטי - "זה לוקח יותר מהרגיל" (ראה 10), וניתן לבטל (Ctrl+C).

#### מסך 4 - Claude Code, state "הוצע fix" [STATE 4]

הפלט המלא, במבנה הקבוע מ-[[#8-ארכיטקטורת-מידע-ia]] עוגן 2.

```
  ⚠ Frame hitch · חומרה גבוהה · ⁨frame #5821⁩ (48.3ms מול תקציב 16.6ms)

  אבחון
  הפריים נתקע כי הוקצו 1.4MB ב-GC בתוך הפריים הזה, מה שגרר
  garbage collection באמצע הפריים. המקור: יצירת רשימה חדשה
  בכל קריאה ב-update loop.

  ראיות
  · frame time   48.3ms  (תקציב 16.6ms)
  · GC alloc     1.4 MB   (תקציב 0)
  · marker דומיננטי:  ⁨EnemyManager.Update -> new List<Enemy>()⁩

  מיקום בקוד
  ⁨Assets/Scripts/EnemyManager.cs:42⁩  (EnemyManager.Update)

  הצעת תיקון
  להקצות את הרשימה פעם אחת ולעשות לה Clear() במקום new בכל פריים:
  - var visible = new List<Enemy>();       (בתוך Update)
  + _visible.Clear();                      (שדה שמוקצה פעם אחת)
  [diff מלא - אם ארוך, מקופל / מוצג בקטע נפרד]

  צעד הבא
  רוצה שאחיל את התיקון, אחדד, או שנדלג על האירוע הזה?
  (אחרי שתחליט - סמן את האירוע בחלון Profilot כ"נסקר / לא בעיה אמיתית")
```
אם אין מיפוי לקוד, סעיף "מיקום בקוד" אומר במפורש: "לא הצלחתי למפות את ה-marker הזה לקוד בריפו - האבחון מבוסס על נתוני הפריים בלבד" (JTBD-3). אם ה-diff ארוך - הוא מקופל או מוצג בקטע נפרד, וסעיף "צעד הבא" חוזר בקצרה בסוף כדי שלא ייבלע מתחת ל-diff (ליטוש לטובת קריאות וצופה לא-מומחה, JTBD-7).

#### מסך 5 - חלון Profilot, state "אין נתונים זמינים" [Flow D]

```
+--------------------------------------------------+
|  Profilot                                   [⚙]  |
+--------------------------------------------------+
|  ○ לא פעיל · אין נתונים                           |   <- חיווי אפור, לא שגיאה
|                                                  |
|     נתוני profiler זמינים רק ב-Play Mode          |
|     (ב-Editor או ב-development build).            |
|                                                  |
|     לחץ Play כדי להתחיל לנטר.        [▶ Play]     |   <- פעולה ישירה אם אפשר
|                                                  |
|     בונה build? ודא development build →           |   <- הסבר ל-NG3, לא הפתעה
+--------------------------------------------------+
```

#### מסך 6 - Claude Code, מסלול דמו (Flow A)

כמו מסך 4 במבנה, אבל מקור הנתונים הוא screenshot/טקסט שהמשתמש סיפק, לא ה-CLI. סעיף "ראיות" מציין שהנתונים נקראו מ-screenshot (ולכן עשויים להיות חלקיים), וסעיף "מיקום בקוד" קיים רק אם Claude הצליח לקרוא את הריפו (best-effort - אין התחייבות, הכרעה 1). אם ה-screenshot חלקי - ראה edge cases ב-10.

**טבלת states מרוכזת:**

| state | משטח | חיווי | מסר מרכזי | פעולה זמינה |
|---|---|---|---|---|
| אין בעיות | Editor | ● ירוק | "אין בעיות שנתפסו" | (כלום, צפייה) |
| נתפס - בשליפה | Editor | ⚠ ענבר + spinner | "שולף נתוני פריים מלא..." | (המתנה; אין אבחון, אין סימון) |
| נתפס - מוכן לאבחון | Editor + Console | ⚠ ענבר | כותרת אירוע + id | בקש אבחון / העתק id / סמן כנסקר |
| מחכה לקלוד | CC | spinner -> ✓/✕ | "מנתח..." | (המתנה, ניתן לבטל) |
| הוצע fix | CC | - | אבחון + תיקון | אמץ / חדד / דלג; ואז סמן כנסקר ב-Editor |
| אין נתונים | Editor | ○ אפור | "רק ב-Play Mode" | Play |
| שגיאה | CC / Editor | ✕ | ראה 10 | retry / הסבר |

### 10. התנהגות אינטראקציה ו-edge cases

#### Loading / המתנה
- **שליפת הפריים** (Editor, על trip): קורית מיד ובלי לחסום את המשחק, ומיוצגת ב-state ביניים "נתפס - בשליפה" (מסך 2א). אם איטית, אין הקפאה גלויה - השליפה רצה ברקע והאירוע עובר ל"מוכן לאבחון" כשהוא persisted.
- **אבחון** (CC): זה ה-progress הארוך - קריאת קוד + LLM. תמיד עם progress גלוי (מסך 3), וה-spinner תמיד מתחלף ב-✓ או ✕. חייב להיות ניתן לביטול (Ctrl+C / "עצור"). שים לב: הרצת ה-CLI עצמה זולה (קריאת קובץ persisted -> JSON, ב-ms); הזמן הוא בשכבת הפרשנות, לא ב-transport.
- ספים: אם האבחון עובר זמן סביר (תפר ל-[[#15-nfrs---ביצועים--אבטחה--scale]] לקביעת הסף), Claude מדווח "זה לוקח יותר מהרגיל" במקום שתיקה.

#### Empty
- **אין אירועים** = state "אין בעיות" (מסך 1), לא מסך ריק. תמיד יש מסר חיובי.
- **תור ריק אחרי סקירה** = חזרה ל-state שקט עם אישור עדין ("האירוע נסקר").
- **דמו בלי קלט** = Claude מבקש בעדינות screenshot או תיאור, לא נכשל.

#### Error
- **CLI נכשל / package לא מותקן** (CC): JSON עם שדה error; Claude מתרגם לעברית: "לא מצאתי את Profilot בפרויקט - בדוק שה-package מותקן ושאתה ב-Play Mode". בלי stack trace גולמי למשתמש.
- **האבחון נכשל אחרי שליפה מוצלחת** (CC): האירוע כבר persisted ב-event store, אבל קריאת הקוד / ה-LLM נכשלה (timeout, ביטול, שגיאת רשת ב-CC). ה-spinner מתחלף ל-✕, וההצעה היא **retry על אותו `eventId` בלי שליפה מחדש** - האירוע לא נעלם מהתור, ואין צורך לחזור ל-Play. "האבחון נכשל; להריץ שוב? `profilot diagnose --id evt_5821_hitch`". זה מבדיל בין כשל transport (לעיל) לכשל פרשנות.
- **screenshot לא קריא / לא של Profiler** (Flow A): Claude אומר מה הוא כן הצליח לקרוא, ומבקש במפורש מה חסר ("אני רואה frame time אבל לא את עץ ה-markers - אפשר screenshot של חלון ה-Hierarchy בפרופיילר?"). לא ממציא מיקום בקוד.
- **marker לא ממופה לקוד**: מטופל מפורשות (JTBD-3) - האבחון יוצא, המיקום אומר "לא מופה".
- **שגיאת שליפת פריים ב-Editor** (state "נתפס - בשליפה" נכשל): החלון מציג state שגיאה דיסקרטי עם כפתור "נסה שוב", והאירוע נשאר בתור. ה-spinner של מסך 2א מתחלף ל-✕, לא נשאר תלוי.

#### Disabled
- **חלון ב-Edit Mode**: כל פעולות האבחון מנוטרלות עם הסבר (מסך 5), לא נעלמות בשקט.
- **כפתור "העתק פקודה" / "סמן כנסקר" במצב "נתפס - בשליפה"**: מנוטרלים עם tooltip "האירוע עוד נשלף". הם נדלקים רק במצב "מוכן לאבחון".
- **כפתור "סמן כנסקר" כשאין אירוע**: מנוטרל עם tooltip "אין אירוע ממתין".

#### Success
- **תיקון אומץ** (CC): אישור ברור + הצעה לחזור ל-Play ולוודא שה-spike נעלם ("רץ שוב? אני אבדוק שזה ירד מתחת לתקציב"). סוגר את הלולאה.
- **אירוע סומן "נסקר / לא בעיה אמיתית"** (Editor, JTBD-8): ה-Editor כותב את ה-feedback ל-event store. "לא בעיה אמיתית" -> M4; "נסקר/עזר" -> M2/M9. סימון בחלון, בלי שיפוטיות, והאירוע יורד מהתור.

#### Destructive actions
- **החלת diff על קוד** היא הפעולה ההרסנית היחידה. ברירת המחדל היא הצעה בלבד (NG2) - לא מוחל בלי אישור מפורש. אם Q4 ([[#6-מדדי-הצלחה]]) ייפתח ל-apply בהסכמה, נדרש אישור מפורש לפני כל כתיבה, ועדיף diff preview לפני החלה. אין "apply all" אוטומטי.
- **ניקוי תור אירועים / שינוי ספי תקציב**: לא הרסני אמיתית, אך שינוי סף שמשתיק התראות צריך feedback ("הסף עלה - אירועים קלים יותר לא יתפסו").

#### Edge cases ספציפיים למוצר
- **מבול אירועים** (אותה בעיה כל פריים): דה-דופ - אירוע חוזר זהה מתאחד לשורת Console אחת עם מונה, לא 60 שורות בשנייה. מנגנון טכני ב-[[#15-nfrs---ביצועים--אבטחה--scale]]; מבחינת UX, המשתמש רואה "Frame hitch ×47" ולא הצפה, ומונה התור סופר אותו כאירוע ייחודי אחד.
- **כמה אירועים שונים בתור**: מוצגים כרשימה קצרה (מסך 2); אבחון "האחרון" (לפי timestamp, ראה Flow B) כברירת מחדל ומסומן ככזה, או לפי id.
- **תוכן ארוך / RTL**: שמות markers, נתיבי קבצים וקוד הם תמיד LTR ומוצגים ב-monospace גם בתוך פסקה עברית RTL (כדי שלא יישברו). **גם `eventId` ו-`frame number` עטופים ב-bidi isolation** (כמו נתיבי קבצים, באמצעות תווי isolate ‎U+2066/U+2069‎ או רכיב נפרד) כדי שלא ייהפכו/יישברו בתוך טקסט עברי. פלט עברי זורם RTL; בלוקי קוד ו-diff נשארים LTR. נתיב קובץ ארוך נחתך מהאמצע (`Assets/.../EnemyManager.cs:42`) עם אפשרות העתקה מלאה. diff ארוך מקופל (מסך 4).
- **מסך קטן / חלון Profilot צר**: החלון עובד מ-~240px רוחב; ספי התקציב מתקפלים, התור נחתך ל-"+N נוספים".
- **פעולות במקביל**: אם המשתמש מבקש אבחון בזמן ש-Play עדיין רץ ותופס אירועים חדשים - האבחון נעשה על snapshot של אירוע ספציפי (id), כך שהוא לא "זז" תחת הרגליים. אירועים חדשים פשוט מצטרפים לתור.

### 11. שיקולי נגישות

נגישות נבנית פנימה, לא מתווספת. שני המשטחים הם text-first, מה שעוזר - אבל יש דגשים.

**מקלדת וניווט (חלון Editor).**
- כל פעולה בחלון Profilot נגישה במקלדת. סדר ה-Tab/focus עוקב אחר היררכיית המידע מ-[[#8-ארכיטקטורת-מידע-ia]]: **חיווי סטטוס -> כותרת האירוע -> "העתק id" -> "העתק פקודה" -> "סמן כנסקר / לא בעיה אמיתית" (JTBD-8) -> רשימת התור -> toggle ספי התקציב (ובתוכו, כשפתוח: שלושת השדות -> "שמור" -> "אפס")**. כפתור "סמן כנסקר" הוא חלק מ-Tab order ולא נשען על עכבר.
- אין מלכודת focus: ה-toggle של ספי התקציב נפתח ונסגר ב-Enter/Space, וה-focus לא נלכד בתוכו; מתוך השדות הפתוחים אפשר להמשיך ב-Tab החוצה.
- העתקת מזהה אירוע, פקודת אבחון, וסימון "נסקר" - כולם זמינים בלי עכבר.

**מקלדת (Claude Code).** ה-CLI טקסטואלי במהותו - כל האינטראקציה היא הקלדה, אין תלות בעכבר. הפעולה ההרסנית היחידה (החלת תיקון) דורשת אישור מפורש בהקלדה, מה שמגן גם נגישותית מפני הפעלה בשוגג.

**ניגודיות וצבע.**
- אסור להישען על צבע בלבד להבחנה בין states. כל state נושא גם **אייקון/צורה וטקסט**: שקט = ● + "פעיל", בשליפה = ⚠ + spinner + "שולף", מוכן לאבחון = ⚠ + "נתפס אירוע", אין נתונים = ○ + "לא פעיל". מי שלא מבחין בין ירוק לכתום עדיין מבדיל לפי הצורה והמילה.
- חיווי האירוע **לא אדום בוהק** - ענבר/כתום, גם כי אדום משדר "תקלה קריטית" וגם לשמירת ניגודיות. יעד ניגודיות טקסט: AA לפחות (4.5:1) מול רקע ה-Editor (גם בערכת כהה וגם בהירה - Unity תומך בשתיהן, ויש לבדוק בשתיהן).
- ב-Claude Code אנו לא שולטים בצבעי ה-terminal; לכן המבנה (כותרות, תוויות סעיף "אבחון"/"ראיות"/"מיקום") נושא את המשמעות, לא הצבע.

**קוראי מסך (screen readers).**
- חלון Editor: לכל חיווי טקסט אלטרנטיבי משמעותי ("Profilot פעיל, אין בעיות" / "Profilot, נתפס frame hitch, אירוע ממתין לאבחון"). כפתור הסגירה מתויג "סמן את האירוע כנסקר או כלא בעיה אמיתית".
- **הכרזת שינוי state** מתבצעת דרך **שורת ה-Console** (live region אמין שתמיד נוכח), ולא דרך חלון Profilot שעשוי לא להיות ב-focus בזמן Play. כשנתפס אירוע, שורת ה-Console (`[Profilot] Frame hitch ... ask Claude Code to diagnose`) היא ההכרזה שה-SR יקרא, באופן מנומס (polite) - מודיע, לא חוטף focus באמצע משחק. אם חלון Profilot כן ב-focus, גם הוא מכריז כ-live region מנומס.
- פלט Claude Code נקרא היטב ב-SR כי הוא טקסט מובנה עם כותרות; הסדר הקבוע מ-IA עוזר ל-SR לנווט.

**יעדי הקשה (hit targets).** כפתורי ההעתקה, הקיפול, ו"סמן כנסקר" בחלון בגודל הקשה סביר (יעד ~24px ומעלה, מותאם לצפיפות ה-Editor), לא אייקונים זעירים.

**Reduced motion.** אין אנימציות מתמשכות ב-state שקט מלכתחילה. ה-spinner ב-Claude Code וב-state "בשליפה" הוא הסטנדרט של ה-CLI/Editor ומסתיים תמיד ב-✓/✕; ב-Editor החיווי לא מהבהב - שינוי state הוא מעבר חד ולא פעימה חוזרת, מה שמכבד reduced motion ולא מסיח בזמן משחק.

**הבהוב והסחה בזמן Play.** מודע לכך שהמשתמש משחק/בודק כשאירוע נתפס. ההתראה דיסקרטית בכוונה (שורת Console אחת + שינוי חיווי שקט), לא pop-up שחוטף focus או עוצר את המשחק - אלא אם המשתמש בחר בכך מפורשות (UX-Q1).

**שאלה פתוחה לבעלים (נגישות):**
- UX-Q3. האם להוסיף קיצור מקלדת גלובלי ב-Editor ל"קפוץ לחלון Profilot / העתק את האירוע האחרון", כדי שמשתמש מקלדת לא יצטרך לאתר את החלון ידנית אחרי כל spike?

### 12. ארכיטקטורת מערכת

הארכיטקטורה נגזרת ישירות משתי הכרעות: האריזה שנסגרה (UPM package + CLI חיצוני, ללא MCP / VS Code extension), והעיקרון הדו-שכבתי מה-brief (tripwire זול מקומי + קריאת LLM רק על אירוע). הניסוח המדויק: **שני assemblies בתוך תהליך Unity (Runtime + Editor) + תהליך CLI נפרד**, ומאגר אירועים אחד על הדיסק שמגשר בין תהליך Unity לתהליך ה-CLI. חשוב לתיקון של ניסוח קודם: ב-Play-in-Editor ה-Runtime וה-Editor הם *שני assemblies באותו תהליך* (תהליך ה-Unity Editor), לא שני processes - אין ביניהם IPC, והערוץ ביניהם הוא in-memory. רק ה-CLI הוא תהליך נפרד (cross-process), ורק שם נכנס ערוץ מבוסס-קבצים.

**שני שכבות בתהליך Unity + תהליך CLI - הרכיבים והגבולות:**

1. **Profilot Runtime (assembly של runtime בתוך ה-UPM package).** רץ בתוך ה-Player loop ב-Play Mode, בתוך תהליך ה-Unity Editor (ב-Play-in-Editor). אחראי בלעדית על שכבת ה-tripwire הזולה: דגימה רציפה של counters דרך `ProfilerRecorder` והערכת היוריסטיקה מקומית מול ספי התקציב. כשהיוריסטיקה מזהה אנומליה היא לא שולפת בעצמה את הפריים המלא (זו פעולת Editor) - היא מסמנת "trip" עם frameIndex ומטא-דאטה מינימלי, ודוחפת אותו לערוץ ה-trip הפנימי (in-memory, ראה למטה). גבול חד: ה-Runtime לא יודע דבר על LLM, על קבצים, או על ה-CLI.

2. **Profilot Editor (assembly של editor בתוך ה-UPM package, `Editor/`).** רץ בתוך אותו תהליך Unity. שלושה תפקידים: (א) צורך את ערוץ ה-trip הפנימי ב-`EditorApplication.update`, ועל trip - שולף את נתוני הפריים המלא דרך `ProfilerDriver` + `HierarchyFrameDataView` / `RawFrameDataView` (ראה [[#14-מודל-נתונים-ו-apis]]), מנרמל ל-JSON, וכותב ל-event store; (ב) חלון ה-EditorWindow (`Window > Profilot`) שמתואר ב-[[#9-מסכים-ו-states-עיקריים]]; (ג) ניהול ה-event store - תור, דה-דופ, retention, וכתיבת `reviewStatus` כשהמשתמש מסמן אירוע (JTBD-8). זהו הצד היחיד שמדבר עם Profiler APIs של ה-Editor ועם ה-event store בכתיבה.

3. **Profilot CLI (תהליך נפרד, מחוץ ל-Unity).** כלי command-line ש-Claude Code מריץ דרך Bash. תפקידו צר בכוונה: לקרוא מה-event store, לבחור אירוע (`--last` או `--id`), ולהדפיס JSON מובנה אחד ל-stdout. ה-CLI **לא** מדבר עם Unity, **לא** קורא ל-LLM, ו**לא** קורא קוד מהריפו. הוא transport טהור: event store -> stdout. הפרשנות (קריאת הקוד, האבחון, הצעת התיקון) קורית כולה ב-Claude Code עצמו, שכבר חי בתוך הריפו - זו הסיבה שאין צורך ב-MCP server. ה-CLI הוא מתאם (adapter) דק; ה-core הוא ה-Runtime + ה-Editor.

**שני ערוצי תקשורת נפרדים - אסור לבלבל ביניהם.**

**ערוץ א' - trip פנימי Runtime -> Editor (in-memory, בתוך תהליך Unity).** זה לא IPC ולא קובץ. ב-Play-in-Editor שני ה-assemblies חולקים תהליך וזיכרון, ולכן ה-Runtime דוחף trip לתור in-memory thread-safe (`ConcurrentQueue<TripSignal>` או callback בודד), וה-Editor צורך אותו בלולאת ה-Editor. **קריטי לתזמון (חוסם שנסגר):** הצריכה חייבת לקרות ב-`EditorApplication.update` או `EditorApplication.delayCall` הראשון אחרי ה-trip - לא "ברקע מתישהו" - כי שליפת הפריים רגישת-זמן (ראה ערוץ ה-ring buffer ב-[[#15-nfrs---ביצועים--אבטחה--scale]]). ה-`TripSignal` נושא `frameIndex` + מטא מינימלי, וה-Editor שולף את הפריים בהקדם האפשרי כדי שלא יידחק מ-ring buffer של ה-Profiler.

**ערוץ ב' - event store Editor -> CLI (מבוסס קבצים, cross-process).** זה הערוץ היחיד שחוצה תהליכים. הצד שכותב (Editor, בתוך תהליך Unity) והצד שקורא (CLI, תהליך נפרד) לא רצים בהכרח בו-זמנית ולא חולקים זיכרון, ולכן הערוץ הוא **event store מבוסס קבצים על הדיסק** - לא socket, לא pipe, ולא IPC חי. הנמקה:
- *פשטות וניתוק זמני.* ה-CLI יכול לרוץ גם כש-Play עצר וגם כשה-Editor נסגר; האירוע כבר persisted. אין צורך ש-Unity יאזין על port או ש-handshake יצליח.
- *התאמה לזרימת Claude Code.* Claude מריץ פקודה קצרה ומקבל stdout - בדיוק מה שקובץ-על-דיסק -> CLI נותן, בלי תלות במצב חי של ה-Editor.
- *עמידות.* קריסת Editor לא מאבדת אירועים שכבר נכתבו; ה-CLI עדיין קורא אותם.

המבנה: ספריית `events/` תחת נתיב מוסכם בתוך הפרויקט (ברירת מחדל `<Project>/Library/Profilot/events/`, מתחת ל-`Library/` כי זה לא-version-controlled ולא אמור להיכנס ל-git; ראה [[#14-מודל-נתונים-ו-apis]] למבנה הרשומה). כל אירוע הוא קובץ JSON אטומי אחד (כתיבה ל-`.tmp` ואז rename, כדי שה-CLI לא יקרא קובץ חצי-כתוב). `latest.json` (index קל המצביע על האירוע האחרון עבור `--last`) **נכתב רק אחרי שקובץ האירוע נכתב במלואו ועבר rename** - הסדר הזה מבטיח שה-CLI לעולם לא יפנה דרך `latest.json` לקובץ אירוע שעוד לא קיים/חלקי. זה פותר את שורת ה-Cross-cutting של ה-event store המשותף.

**מענה לשורת ה-Cross-cutting על כפתור Play (UX מסך 5).** הפעלת Play Mode מתוך EditorWindow אכן טריוויאלית טכנית - `EditorApplication.isPlaying = true`. הכפתור "▶ Play" במסך 5 ישים ונשאר. אין צורך בנפילה להנחיה טקסטואלית. סוגר את השורה ביומן.

**זרימת נתונים מקצה לקצה (Flow B, המוצר האמיתי):**

```
[Play Mode]  ┌─────────── תהליך Unity (שני assemblies) ───────────┐
  Runtime: ProfilerRecorder דוגם counters כל פריים
     │  היוריסטיקה מקומית מול ספי תקציב
     ▼ (anomaly? -> push TripSignal)
  ערוץ trip פנימי (in-memory queue/callback) ── אותו תהליך, בלי IPC
     ▼ נצרך ב-EditorApplication.update הראשון (רגיש-זמן!)
  Editor: ProfilerDriver/HierarchyFrameDataView שולף פריים מלא מיד
     │  נרמול -> JSON, דה-דופ, הוספה לתור
     ▼ כתיבה אטומית (event*.json ואז latest.json)
  └────────────────────────────────────────────────────────────┘
  Event store on disk  (Library/Profilot/events/*.json + latest.json)
     ▲ קריאה (--last / --id)  ── cross-process, מבוסס קבצים
     │
  CLI (תהליך נפרד): בוחר אירוע -> מדפיס JSON ל-stdout
     ▲ הרצה דרך Bash
     │
  Claude Code: קורא JSON + קורא קוד מהריפו -> אבחון + הצעת fix
```

**איפה חי ה-state.** ה-state האמיתי והמתמשך הוא ה-event store על הדיסק - מקור האמת היחיד שגם ה-Editor וגם ה-CLI מסכימים עליו. ה-Runtime מחזיק רק state נדיף (ערכי ה-recorders האחרונים, חלון ההיוריסטיקה, ותור ה-trip ה-in-memory עד שה-Editor צורך אותו). חלון ה-Editor הוא view בלבד מעל ה-event store ומעל ה-Runtime; הוא לא מקור אמת - **למעט** כתיבת `reviewStatus` כשהמשתמש מסמן אירוע (JTBD-8), שזו הכתיבה היחידה של ה-Editor ל-store מעבר לכתיבת האירוע עצמו. Claude Code הוא stateless ביחס לכלי - הוא קורא JSON טרי בכל הרצה.

**הערה על מסלול הדמו (Flow A).** מסלול הדמו עוקף את כל שלושת הרכיבים: אין Runtime, אין Editor, אין CLI. המשתמש מספק screenshot/טקסט ישירות ל-Claude Code, ש-Claude קורא ומאבחן מול הקוד בריפו. זה למה הדמו עובד "אפס אינטגרציה" - הוא לא תלוי ב-package כלל. המשמעות הארכיטקטונית: מבנה הפלט של האבחון ([[#9-מסכים-ו-states-עיקריים]] מסך 4) חייב להיות זהה בשני המסלולים, כי בדמו אין סכמת JSON אוכפת אותו - רק הנחיית הפרויקט (ראה [[#13-סטאק-טכנולוגי-ונימוק]]).

### 13. סטאק טכנולוגי ונימוק

**Profilot Runtime + Editor (ה-UPM package)**
- **C# / .NET (Unity scripting)** - חובה, זו שפת ה-Editor וה-runtime ב-Unity. אין בחירה.
- **`Unity.Profiling.ProfilerRecorder`** (Unity 2020.2+) - ה-tripwire. נבחר כי הוא ה-API הרשמי והזול לקריאת counters בזמן ריצה, ועובד ב-Editor וב-Players כאחד. ראה אילוץ הגרסה למטה.
- **`UnityEditorInternal.ProfilerDriver` + `UnityEditor.Profiling.HierarchyFrameDataView` / `RawFrameDataView`** - שליפת הפריים המלא בצד ה-Editor. זה ה-API שה-Profiler window עצמו משתמש בו. אזהרה: `ProfilerDriver` הוא internal/undocumented ויש דיוני deprecation - מתועד כסיכון ב-[[#16-סיכונים-ומיטיגציה]].
- **`FrameTimingManager`** - timings של CPU/GPU ברמת פריים, ב-overhead נמוך. דורש `CaptureFrameTimings()` בכל פריים ואז `GetLatestTimings()`; ב-Editor/dev זמין, ב-release דורש הפעלת Frame Timing Stats ב-Player settings (לא רלוונטי לנו, ראה NG3).
- **UI Toolkit (UIElements)** לחלון ה-Editor - הסטנדרט המודרני ל-EditorWindow ב-Unity, נגיש למקלדת ול-SR מתוך הקופסה (תומך בדרישות [[#11-שיקולי-נגישות]]), ונקי יותר מ-IMGUI. ברף 6000.3 (הכרעה 4) UI Toolkit בוגר ב-Editor, ולכן הוא הבחירה ואין צורך ב-fallback ל-IMGUI.
- **System.Text.Json לא זמין בכל גרסאות Unity** - לכן סריאליזציה ל-event store דרך **`UnityEngine.JsonUtility`** למבנים פשוטים, או Newtonsoft (`com.unity.nuget.newtonsoft-json`, package רשמי) למבנים מקוננים כמו עץ markers. נטייה ל-Newtonsoft כי עץ ה-markers מקונן ו-`JsonUtility` חלש בקינון/מערכים פולימורפיים.

**Profilot CLI**
- **Node.js, מופץ דרך npm (הכרעה 3).** ה-CLI הוא transport טהור (pass-through JSON: קורא קובץ אירוע מה-store, מדפיס ל-stdout). מכיוון שהוא לא מפרסר/בונה את הסכמה אלא רק מעביר אותה, שיתוף DTOs עם ה-package כמעט לא רלוונטי, וקלות ההפצה דרך npm (`npx profilot ...` / התקנה גלובלית, ה-brief ציין ש-npm פנוי) מכריעה. סוגר את A-Q1.
- הפלט: **JSON יחיד ל-stdout**, exit code 0 גם על "no_data" / "error" (השגיאה בתוך ה-JSON כשדה `status`), כדי ש-Claude Code יוכל לפרסר תמיד במקום ליפול על non-zero exit. שגיאות תפעוליות אמיתיות (ארגומנט לא חוקי) - non-zero + JSON ל-stderr.

**שכבת הפרשנות (Claude Code)**
- אין כאן "טכנולוגיה" שאנחנו בונים מעבר ל**הנחיית פרויקט** (קובץ הנחיות שמתווסף ב-B0, ראה [[#7-user-flows-ומסעות]]) שמלמדת את Claude: מתי להריץ `profilot diagnose --last`, איך לפרסר את ה-JSON, ובאיזה מבנה קבוע להחזיר את האבחון ([[#9-מסכים-ו-states-עיקריים]] מסך 4). זה גם מה שאוכף את אחידות הפלט בין מסלול הדמו למסלול ה-live.
- מודל ה-LLM הוא Claude (דרך Claude Code שכבר אצל המשתמש) - אין לנו עלות API ישירה ואין ניהול מפתח מצדנו; העלות היא של המשתמש, מה שמפשט מאוד את [[#15-nfrs---ביצועים--אבטחה--scale]] (אין secrets לנהל).

**אילוצי פלטפורמה וגרסה (נסגר - הכרעה 4):**
- **רף מינימלי רשמי: Unity 6000.3** (עודכן על ידי הבעלים 2026-07-02; קודם 2022.3 LTS). זו ההכרעה המחייבת וזו הגרסה שעליה המוצר פותח ואומת חי. בגרסה זו `ProfilerRecorder` בוגר, `HierarchyFrameDataView` / `RawFrameDataView` יציבים, ו-UI Toolkit ב-Editor בשל.
- לידיעה היסטורית: `ProfilerRecorder` קיים כבר מ-Unity 2020.2, אך אנחנו לא תומכים מתחת ל-6000.3 כדי לא לפצל מאמץ בדיקה ולתמוך רק בגרסה שאומתה בפועל.

> הערה: A-Q1 (שפת ה-CLI) נסגרה - Node.js + npm (הכרעה 3), ראה למעלה. Q1 (רף Unity) נסגרה - 6000.3 (הכרעה 4, עודכן 2026-07-02). אין כרגע שאלות טכניות פתוחות בסעיף זה.

### 14. מודל נתונים ו-APIs

**מקור הנתונים (Editor APIs, על trip).**
- שכבת ה-tripwire קוראת counters דרך `ProfilerRecorder` (ערך אחד או חלון של ערכים אחרונים per counter). הסט המינימלי: `Main Thread` frame time, `GC Allocated In Frame`, `Draw Calls Count`, `SetPass Calls Count`, `Triangles Count`, `Batches Count`, ו-Total Reserved/Used Memory. **זמינות counter ספציפי אינה מובטחת** (משתנה לפי גרסת Unity, render pipeline ו-platform), ולכן כל counter מתגלה דרך `ProfilerRecorderHandle.GetAvailable` ונבדק עם `ProfilerRecorder.Valid` לפני קריאה; counter שלא קיים פשוט נשמט מהסט הפעיל ולא מפיל את ה-tripwire. ה-API עצמו זמין גם ב-release, אבל זמינות counter בודד עדיין משתנה (תיקון לניסוח R3 הגורף).
- על trip, ה-Editor פותח את הפריים: `ProfilerDriver.GetHierarchyFrameDataView(frameIndex, threadIndex, sortColumn, sortAscending, collapse)` לעץ ה-markers ההיררכי המצרפי (זמן + alloc per marker כולל ילדים), או `ProfilerDriver.GetRawFrameDataView(frameIndex, threadIndex)` לגישה גולמית per-sample. ה-frameIndex נע בין `ProfilerDriver.firstFrameIndex` ל-`lastFrameIndex`.
- timings של CPU/GPU מ-`FrameTimingManager.GetLatestTimings` (אחרי `CaptureFrameTimings` בכל פריים ב-Runtime).

**רשומת אירוע (event record) - הסכמה ש-Editor כותב ו-CLI קורא.** זה החוזה המשותף שפותר את שורת ה-Cross-cutting. מבנה לוגי (לא קוד סופי):

```
ProfilotEvent {
  schemaVersion   // למשל "1" - לעמידות מול שינויי סכמה בין גרסאות package/CLI
  eventId         // "evt_<frame>_<type>"  (תואם ל-UX מסך 2)
  status          // "ok" | "no_data" | "error"
  reviewStatus    // "open" | "reviewed" | "not_a_real_issue"  (JTBD-8)
                  // ברירת מחדל "open"; נכתב ע"י ה-Editor כשהמשתמש מסמן בחלון.
                  // "not_a_real_issue" -> קלט ל-M4; "reviewed" -> M2/M9. ה-CLI קורא בלבד.
  stale           // bool - true אם האירוע ממ-session שכבר נסגר (ראה retention ב-NFR)
  capturedAt      // timestamp
  sessionId       // מזהה session ריצה, לסימון stale ולשבירת שוויון
  unityVersion
  frameIndex
  requestedFrameIndex   // ה-frameIndex שה-trip ביקש; אם שונה מ-frameIndex ->
                        // הפריים נדחק מ-ring buffer ונשלף הקרוב ביותר (ראה NFR, מירוץ ring buffer)
  frameIndexDelta       // requestedFrameIndex - frameIndex (0 = נשלף בדיוק המבוקש)
  trigger {       // מה הדליק את ה-tripwire
    type          // "frame_hitch" | "gc_spike" | "draw_calls" | ...
                  // NG5/physics: hitch שמקורו physics נתפס כ-frame_hitch גנרי
                  // (אין trigger type ייעודי ל-physics באיטרציה זו); ה-markerTree
                  // עשוי לחשוף Physics.Simulate כ-marker דומיננטי, והאבחון רשאי
                  // לומר "המקור ב-physics, מחוץ לסקופ הניתוח הנוכחי".
    severity      // נגזר מ-value מול budget
    metric, value, budget
  }
  counters {      // הסיכום הזול מה-Runtime; שדה חסר = counter לא היה Valid בגרסה/פלטפורמה
    frameTimeMs, cpuFrameTimeMs, gpuFrameTimeMs,
    gcAllocBytes, drawCalls, setPassCalls, triangles, batches,
    memReservedBytes, memUsedBytes
  }
  markerTree[]    // עץ ה-HierarchyFrameDataView המנורמל:
                  // { name, selfTimeMs, totalTimeMs, gcAllocBytes, calls, children[] }
                  // נחתך כדי לא לנפח את הקובץ - אסטרטגיית חיתוך למטה
  topMarkers[]    // N ה-markers הדומיננטיים (לפי self time / alloc) - לנוחות Claude
  callstacks[]?   // אם enableAllocationCallstacks פעיל: backtrace ל-GC.Alloc (אם זמין)
  dedup {         // count, firstSeen, lastSeen - לאיחוד אירוע חוזר
    count, firstSeenFrame, lastSeenFrame
  }
}
```

**אסטרטגיית חיתוך `markerTree`.** עץ ה-markers הגולמי יכול להכיל אלפי צמתים; שמירתו במלואו תנפח את הקובץ ותאט את הנרמול. כלל החיתוך: **שומרים כל ענף ש-`selfTimeMs` שלו מעל סף יחסי (ברירת מחדל ~1% מזמן הפריים) או ש-`gcAllocBytes` שלו מעל 0; משם שומרים top-N ילדים לפי self time בכל רמה (ברירת מחדל N=8), ומגבילים עומק (ברירת מחדל 6).** צמתים שנחתכו מתקפלים ל-marker סינתטי אחד (`"<other> ×K, Σself=…ms"`) כדי שלא ייעלם זמן מהתמונה. `topMarkers` נגזר מהעץ המלא *לפני* החיתוך (לפי self time / alloc יורד) כדי שה-N הדומיננטיים תמיד נשמרים גם אם נחתכו מהעץ ההיררכי. הספים האלה ניתנים לכוונון ונכללים בכיול שלב 3.

**מיפוי marker -> קובץ/שורה (הליבה של JTBD-3, והחלק הכי לא-ודאי).** אין ב-Unity API ישיר ואמין שמחזיר "marker X = שורה Y בקובץ Z". מה שכן יש, בסדר יורד של איכות:
1. **callstacks של allocations** - כאשר `Profiler.enableAllocationCallstacks` / כפתור "Call Stacks" פעיל, ל-GC.Alloc samples מצורף backtrace שכולל שמות מתודות (ולעיתים קובץ/שורה). זה האות הטוב ביותר לבעיות GC. אזהרה מאומתת: ה-backtrace לא תמיד מצביע על השורה המדויקת, ולעיתים מצביע על frame שגוי.
2. **שם ה-marker עצמו** - markers מובנים נושאים שם מתודה (`EnemyManager.Update`), ו-`ProfilerMarker` מותאמים אישית נושאים שם שהמפתח נתן. שם המתודה הוא מחרוזת חיפוש מצוינת בריפו.
3. **הסקה של Claude** - Claude Code מקבל את שם ה-marker + ה-backtrace (אם יש) **כ-hints**, ומבצע את המיפוי בפועל על ידי grep/קריאה בריפו (הוא כבר בתוך הפרויקט). כלומר המיפוי הסופי הוא אחריות שכבת הפרשנות, לא ה-package. ה-package מספק את ה-hints הכי טובים שיש; Claude מאמת מול הקוד האמיתי.

מסקנה ארכיטקטונית: ה-event record **לא** מתחייב על `file:line` מדויק. הוא מספק `markerTree`, `topMarkers`, ו-`callstacks?` כ-evidence, ו-Claude אחראי על ההצמדה לקוד - כולל הדרישה מ-JTBD-3 לומר מפורשות כשלא נמצא מיפוי. זה גם למה M3 (דיוק מיפוי) הוא יעד שאפתני שמוגדר כ"ייכויל".

**ממשק ה-CLI (החוזה כלפי Claude Code).**
- `profilot diagnose --last` -> JSON של האירוע האחרון מה-event store.
- `profilot diagnose --id <eventId>` -> JSON של אירוע ספציפי (תומך ב-snapshot היציב מ-[[#10-התנהגות-אינטראקציה-ו-edge-cases]] "פעולות במקביל").
- `profilot list` -> רשימת אירועים בתור (id, type, severity, time, `reviewStatus`, `stale`) - תומך בתצוגת התור ומאפשר ל-Claude/המשתמש לדלג על אירועים שכבר נסקרו או מ-session קודם.
- `profilot status` -> האם package מזוהה, האם Play רץ, כמה אירועים בתור. משרת את מצב "no_data" מ-Flow D.
- כל הפקודות: JSON ל-stdout, exit 0, השגיאה כשדה `status` בתוך ה-JSON (ראה [[#13-סטאק-טכנולוגי-ונימוק]]).

**מסלול הדמו (Flow A) - אין סכמה אוכפת.** בדמו אין event record; הקלט הוא screenshot/טקסט. לכן אחידות הפלט מובטחת רק על ידי הנחיית הפרויקט שמגדירה את אותו מבנה אבחון. זו נקודת תורפה מודעת (מתועדת ב-[[#16-סיכונים-ומיטיגציה]]).

### 15. NFRs - ביצועים / אבטחה / scale

**ביצועים - overhead הניטור (קריטי, קלט ל-M8).**
- שכבת ה-tripwire היא ה-NFR החשוב ביותר: היא רצה כל פריים ב-Play Mode, ואסור שתורגש. **היעד הוא §6 M8: מתחת ל-0.5ms לפריים בזמן שקט** (כ-3% מתקציב 16.6ms). מספר ה-0.1ms שהוזכר קודם הוא *שאיפה לא-מבוססת* - הוא מסומן כ**יעד-לאימות שיימדד בשלב 1**, לא כעובדה. אם המדידה תראה שהדגימה יקרה מהצפוי, fallback מתוכנן: **דגימה בתדירות נמוכה** (למשל כל k פריימים, או דגימת counters יקרים בלבד בתת-קצב) במקום כל פריים, תוך שמירה על תפיסת hitch בודד דרך frame time שכן נקרא בכל פריים.
- **עלות `FrameTimingManager` נכללת במפורש ב-budget.** `CaptureFrameTimings()` נקרא בכל פריים ב-Runtime (לפני `GetLatestTimings`), ולכן עלותו היא חלק מ-overhead ה-tripwire ונמדדת תחת אותו יעד M8 - לא "חינם". אם תימצא יקרה, היא מועמדת הראשונה ל-fallback של דגימה בתדירות נמוכה / השבתה אופציונלית (CPU/GPU split יורד ל-best-effort).
- אין הקצאות בלולאת הדגימה (recorders מוקצים פעם אחת ב-setup, אפס GC alloc ב-hot path - אחרת הכלי בעצמו יוצר את הבעיה שהוא מחפש).
- הפעולה היקרה (`ProfilerDriver` + שליפת `HierarchyFrameDataView` + נרמול + כתיבת קובץ) קורית **רק על trip**, בצד ה-Editor, ולא בלולאת ה-Runtime. גם אז היא לא חוסמת את המשחק (תפר ל-[[#10-התנהגות-אינטראקציה-ו-edge-cases]]: השליפה רצה ולא מקפיאה). חיתוך `markerTree` (ראה [[#14-מודל-נתונים-ו-apis]]) שומר על גודל קובץ סביר וזמן נרמול קצר.
- ספי ברירת מחדל (קלט ל-Q2, החלטת בעלים, כאן רק הצעה טכנית): frame budget 16.6ms (60fps), `gcAllocBytes > 0` בפריים כסף בסיסי ל-GC (ניתן לכוונון כי לא כל alloc הוא בעיה), spike ב-draw calls יחסי ל-baseline נע ולא סף מוחלט.

**מירוץ ה-ring buffer בין trip לשליפה (קריטי - חוסם שנסגר).**
- ה-Unity Profiler שומר חלון מוגבל של פריימים אחרונים (כ-300 כברירת מחדל; הטווח החי הוא `ProfilerDriver.firstFrameIndex`..`lastFrameIndex`). יש מירוץ: בין הרגע שה-Runtime מסמן trip על `frameIndex` לרגע שה-Editor שולף, הפריים עלול להידחק מה-ring buffer - וזה דווקא הנתיב הרגיל *תחת עומס*, בדיוק כשיש hitches. retry "מתישהו ברקע" חסר ערך כי הפריים כבר אבד.
- מיטיגציה (מחייבת):
  1. **שליפה מיידית, רגישת-זמן.** ה-Editor חייב לצרוך את ה-trip ולשלוף את הפריים ב-`EditorApplication.update` / `delayCall` הראשון אחרי ה-trip (ראה ערוץ א' ב-[[#12-ארכיטקטורת-מערכת]]), לא בעבודת רקע דחויה. הפעולה מסומנת כרגישת-זמן בקוד ובתיעוד.
  2. **תיעוד דלתא.** אם הפריים המבוקש כבר אינו בטווח, שולפים את הקרוב ביותר בטווח ורושמים `requestedFrameIndex`, `frameIndex`, ו-`frameIndexDelta` ברשומה ([[#14-מודל-נתונים-ו-apis]]); דלתא גדולה היא סימן ל-Claude (וב-state שגיאה אם הפריים אבד לגמרי).
  3. **הגדלת history buffer.** ממליצים/מגדירים הגדלת `ProfilerDriver` history (`Profiler.maxUsedMemory` / הגדלת מספר הפריימים השמורים ב-Profiler) ברף הסביר, כדי להרחיב את חלון ההזדמנות לשליפה לפני דחיקה. הערך המדויק נכלל בכיול שלב 1/3.
  4. אם בכל זאת אבד -> `status:"error"` עם הסבר, האירוע נשאר בתור עם אפשרות retry (אבל retry יצליח רק אם הפריים עדיין בטווח - לכן 1-3 הם העיקר).

**עלות LLM וניהול תדירות (קריטי, JTBD-6, M6, M7).**
- העלות נשלטת בעיקרה ארכיטקטונית: **אין קריאת LLM אוטומטית בכלל**. הכלי לא מתקשר עם Claude מיוזמתו; המשתמש (או Claude לבקשתו) מריץ את ה-CLI, ושכבת הפרשנות היא Claude Code של המשתמש. כלומר M6 ("אפס קריאות LLM בזמן שקט") מתקיים בינארית מעצם התכנון - ה-CLI הוא קורא-קובץ, הוא לא מתקשר החוצה.
- שליטת תדירות בשכבת האירועים (כדי שלא יצטבר מבול אירועים שיגרור מבול אבחונים): **דה-דופ** - אירוע מאותו `type` על אותו "אזור" בתוך חלון זמן קצר מתאחד לרשומה אחת עם `dedup.count` עולה (תומך ב-"Frame hitch ×47" מ-[[#10-התנהגות-אינטראקציה-ו-edge-cases]]). **הגדרת "אזור" לפי שלב:** בשלב 1 (counters-only, בלי markerTree) אין marker דומיננטי, ולכן ה-dedup key הוא **`type` בלבד** (כל frame_hitch מתאחד לאירוע אחד). משלב 2 ואילך, כשיש markerTree, ה-key מתעדן ל-`type` + ה-marker הדומיננטי (`topMarkers[0].name`), כך ש-hitch מ-`EnemyManager.Update` ו-hitch מ-`Physics.Simulate` נספרים כשני אירועים נפרדים. **Rate limit / cooldown** per trigger type - אחרי trip, cooldown קצר לפני trip נוסף מאותו סוג. **תקרת תור** - חלון מחזיק עד N אירועים (ברירת מחדל מוצעת ~20); מעבר לכך, החדש דוחק את הישן ביותר שלא נסקר (`open`), או מתאחד. זה פותר את UX-Q2.

**Scale / retention (תיקון מול עמידות בקריסה).**
- ה-event store מוגבל אך **לא נמחק בגורף ב-startup**. הסכנה בניסוח הקודם: מחיקת אירועים ב-startup הייתה מוחקת בדיוק אירועים ששרדו קריסה (הערך הגדול ביותר של persistence). לכן הכלל המתוקן:
  - אירועים מ-session קודם **לא נמחקים ב-startup** - הם **מסומנים `stale:true`** (וה-`reviewStatus` שלהם נשמר). כך `--last` ו-`list` עדיין מנגישים אותם לפני כל מחיקה, ואירוע ששרד קריסה זמין לאבחון.
  - מחיקה היא **לפי גיל ולפי תקרה בלבד**, ורק על אירועים שכבר נסקרו או שעברו את גיל ה-retention; אירוע `open` שעדיין לא נסקר לא נמחק רק כי התחיל session חדש.
  - תקרת התור (~20, ראה למטה) דוחקת את הישן-ביותר-שלא-נסקר; אירוע שנסקר נמחק מוקדם יותר. אין צמיחה בלתי-מוגבלת על הדיסק. הגודל הצפוי קטן (JSON של פריים בודד, קילובייטים עד עשרות KB אחרי חיתוך ה-markerTree).
- **סימון stale.** אירוע מ-`sessionId` שאינו ה-session הפעיל מסומן `stale:true`; ה-CLI מחזיר אותו עם הדגל, ו-Claude/החלון יכולים לציין "אירוע מ-session קודם". זה מונע אבחון שמתייחס לפריים שכבר לא רלוונטי לקוד הנוכחי בלי להתריע.
- אין ריבוי משתמשים, אין שרת, אין concurrency אמיתי מעבר ל-Editor-כותב / CLI-קורא על אותו קובץ - וזה נפתר בכתיבה אטומית (tmp + rename, `latest.json` אחרון), ובכך שה-CLI הוא read-only על ה-store.

**אבטחה ופרטיות.**
- **אין secrets לנהל מצדנו** - אין מפתח API ל-LLM (הפרשנות היא Claude Code של המשתמש). זו פשטות אבטחתית משמעותית; אין מה להדליף.
- ה-event store יושב תחת `Library/` ולא נכנס ל-git (מתועד גם ב-`.gitignore` של ה-package אם רלוונטי). הוא מכיל נתוני profiler ושמות markers/מתודות - לא סודות, אבל לא צריך להגיע ל-version control.
- ה-CLI לא מבצע פעולות רשת ולא כותב מחוץ לתחום הפרויקט. החלת תיקון על קוד (הפעולה ההרסנית היחידה, [[#10-התנהגות-אינטראקציה-ו-edge-cases]]) נעשית כולה בצד Claude Code עם אישור מפורש - לא ב-package ולא ב-CLI.
- נתוני הקוד שמגיעים ל-LLM: הם נשלחים על ידי Claude Code של המשתמש לפי שיקול דעתו והגדרותיו, כמו כל שימוש אחר ב-Claude Code. הכלי לא יוצר ערוץ exfiltration חדש.

**אמינות / failure modes.**
- קריסת Editor: אירועים שכבר נכתבו אטומית שורדים; ה-CLI קורא אותם גם אחרי קריסה.
- קובץ event חצי-כתוב: נמנע על ידי tmp + rename; ה-CLI לעולם לא רואה קובץ חלקי.
- `ProfilerDriver` מחזיר frame לא תקין / frameIndex מחוץ לטווח (הפריים כבר נדחק מ-ring buffer של ה-Profiler): מטופל לפי מנגנון המירוץ למעלה - אם יש פריים קרוב בטווח שולפים אותו ומתעדים `frameIndexDelta`; אם אבד לגמרי, נכתב event עם `status:"error"` והודעה, האירוע נשאר בתור עם אפשרות retry ([[#10-התנהגות-אינטראקציה-ו-edge-cases]]). הקו ההגנתי האמיתי הוא השליפה המיידית + הגדלת ה-buffer, לא ה-retry.
- ה-CLI לא מוצא event store / package לא מותקן: `status:"no_data"` / `"error"` ב-JSON, Claude מתרגם לעברית (תפר ל-[[#10-התנהגות-אינטראקציה-ו-edge-cases]]).

### 17. Roadmap בשלבים

העיקרון: כל שלב מספק ערך עצמאי ובר-בדיקה, ושום שלב מוקדם לא תלוי בשלב מאוחר. שלב 0 הוא האימות הזול ביותר מקצה-לקצה של ההנחה המסוכנת ביותר (שקלוד מסוגל בכלל לאבחן מנתוני profiler) - והוא לא נוגע בחלק הטכני הכי שביר (R1).

**שלב 0 - דמו / screenshot (Flow A) + spike טכני ל-R1. מאמת: "קלוד מאבחן profiler באמת" ובמקביל מוודא ש-`ProfilerDriver` שליף.**
- היקף עיקרי (Flow A): **קובץ הנחיית הפרויקט התבניתי** (deliverable מוגדר, הכרעה 7) שמלמד את Claude Code לקבל screenshot/טקסט של ה-Profiler, לקרוא קוד מהריפו, ולהחזיר אבחון במבנה הקבוע ([[#9-מסכים-ו-states-עיקריים]] מסך 4/6). המשתמש מעתיק/מקשר אותו ל-CLAUDE.md ידנית. אפס קוד Unity, אפס CLI, אפס package.
- **spike טכני מקביל ומנותק (מוריד את R1 מוקדם):** ניסוי קטן ועצמאי שבודק שליפת `HierarchyFrameDataView` מ-**פריים יחיד** דרך `ProfilerDriver` ב-Unity 6000.3 - בלי tripwire, בלי event store, בלי שום אינטגרציה. רק לאמת שה-API ה-internal הזה אכן מחזיר עץ markers שמיש מול הרף שנקבע (הכרעה 4). זה מאמת את R1 (ההנחה הטכנית השברירית ביותר) לפני ששלב 2 נשען עליו, בלי לדחות את הדמו. הספייק לא חלק מהמוצר - תוצרו הוא ידע (כן/לא + צורת ה-API), ואופציונלית שלד ל-`IFrameDataSource`.
- תלויות: הדמו תלוי רק ב-Claude Code עצמו. הספייק תלוי רק ב-Unity 6000.3. שניהם בלתי-תלויים זה בזה ובשאר השלבים.
- קריטריון יציאה בר-בדיקה: (א) על סט תרחישי screenshot ידועים (GC כל פריים, draw calls, hitch), Claude מחזיר אבחון נכון במבנה הקבוע ב-90% מההרצות (קלט ל-M1, M2) - מייצב את מבנה הפלט (מיטיגציית R6) לפני ש-live נבנה; (ב) הספייק מראה שאפשר לשלוף עץ markers שמיש מפריים יחיד ב-`ProfilerDriver` (R1 מאומת חיובי), או מסמן דגל אדום מוקדם אם לא - לפני שלב 2.

**שלב 1 - tripwire מקומי בלבד, בלי שליפת פריים מלא ובלי CLI. מאמת: "אפשר לתפוס אנומליה בזול בזמן ריצה".**
- היקף: Runtime assembly עם `ProfilerRecorder` (גילוי דרך `GetAvailable`/`Valid`) + `FrameTimingManager` + היוריסטיקה + ספי תקציב, וחלון Editor בסיסי שמראה את states "אין בעיות" / "נתפס spike" ([[#9-מסכים-ו-states-עיקריים]] מסך 1, 2, 5) ושורת Console. האירוע נכתב ל-event store עם counters בלבד (בלי markerTree עדיין; dedup לפי `type` בלבד בשלב זה - ראה [[#15-nfrs---ביצועים--אבטחה--scale]]).
- תלויות: רק `ProfilerRecorder` (זמין מ-2020.2, נבדק מול רף 6000.3). לא תלוי ב-`ProfilerDriver`.
- קריטריון יציאה: בהרצה נקייה - אפס אירועים; **overhead ה-tripwire (כולל `CaptureFrameTimings`) נמדד בפועל ומאומת מתחת ליעד M8 (מתחת ל-0.5ms/פריים)** - זו המדידה שמאשרת או מפריכה את שאיפת ה-0.1ms, ואם נכשל מפעילים את fallback הדגימה בתדירות נמוכה ([[#15-nfrs---ביצועים--אבטחה--scale]]). בתרחיש בעיה מתוכנן - אירוע נתפס ונכתב לקובץ אטומית (`latest.json` אחרון), עם דה-דופ פעיל (M4, M5, M6). חלון ה-Editor משקף נכון את ה-states וכותב `reviewStatus` על סימון ידני.

**שלב 2 - שליפת פריים מלא + CLI + מסלול live מלא (Flow B). מאמת: "הלולאה השלמה Editor -> CLI -> Claude עובדת".**
- היקף: שליפת `HierarchyFrameDataView` / `RawFrameDataView` על trip (ערוץ trip פנימי -> `EditorApplication.update`, שליפה מיידית רגישת-זמן, ראה [[#12-ארכיטקטורת-מערכת]] ו-[[#15-nfrs---ביצועים--אבטחה--scale]]) ונרמול ל-`markerTree` חתוך + `topMarkers` (+`callstacks?`), ה-CLI ב-**Node.js/npm** (הכרעה 3) עם `diagnose --last/--id`, `list`, `status`, והנחיית הפרויקט המורחבת שמלמדת את Claude להריץ את ה-CLI. dedup מתעדן ל-`type` + marker דומיננטי. הצמדת marker -> קוד דרך ה-hints + grep של Claude.
- תלויות: בונה על שלב 1 (ה-event store וה-Runtime כבר קיימים) ועל שלב 0 (מבנה הפלט כבר מיוצב, ו-R1 כבר אומת בספייק). כאן נכנס `ProfilerDriver` מאחורי ה-abstraction של R1. (A-Q1 כבר סגורה - Node, הכרעה 3; אין מה להכריע באמצע השלב.)
- קריטריון יציאה: מ-trip ב-Play Mode, המשתמש מבקש מ-Claude "תאבחן את האחרון", Claude מריץ את ה-CLI, מקבל JSON עם עץ markers, ומחזיר אבחון + הצעת fix במבנה הקבוע (M3 נמדד כאן לראשונה). מצבי no_data / error מטופלים נכון.

**שלב 3 - ליטוש, כיול וכוונון (לא פיצ'רים חדשים).**
- היקף: כיול ספי ברירת המחדל מול שימוש אמיתי (Q2), כוונון retention/rate-limit (UX-Q2), כיול M3/M4, ליטוש נגישות החלון ([[#11-שיקולי-נגישות]]), אופציונלית קיצור מקלדת גלובלי (UX-Q3) ופתיחת Q4 ל-apply בהסכמה אם יוחלט.
- קריטריון יציאה: M4 (false positives) מתחת ל-15% במדגם, M7 (עלות) בתוך הסף שייקבע (Q5), M9 (מעבר דמו->live) נמדד.

**מחוץ ל-roadmap הזה (תואם [[#5-scope---בפנים--בחוץ]] ו-NG5):** GPU profiling לעומק, physics לעומק, היסטוריית אירועים מתמשכת / דשבורד, שיתוף צוותי, release-build telemetry. כל אלה הרחבות עתידיות שהארכיטקטורה הדו-שכבתית + ה-event store + ה-CLI כבר תומכים בהן בלי שינוי מבני (למשל היסטוריה = הסרת ה-retention cap; GPU = counters נוספים ב-`ProfilerRecorder`).
